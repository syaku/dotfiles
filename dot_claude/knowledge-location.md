規範・知識は、それが効くべき範囲に置く。新しく規範・知識・好み・運用ノウハウを永続化するときは、**どの器に置くかをまず判断する**。

- 器の選択表（auto memory / paths なし rules / paths 付き rules / local-*.md / CLAUDE.md / CLAUDE.md からの @import / skill）・paths glob の書き方・local 運用の詳細は **`/knowledge-placement` skill** を参照。
- auto memory に行動規範を溜めない（recalled memory は context 扱いで強制力が弱く、常時ロードの rules の方が強い）。memory は事実・人物像・タスク状態のみ。書く際の書き口は [[agent-behavior]] 「auto memory の書き口」節を参照。

### CLAUDE.md と paths なし rules の境界

両者は常時ロードでトークンコストが同じ。分けるのは**文書形式**であり「知識か規範か」ではない。

- 抜いたら周囲の地の文が壊れる → **CLAUDE.md**（通読される一貫文書。順序と文脈が意味を担う。環境の入口・索引・他の器へのポインタ）。
- 単独で足し引きできる自己完結した規範 → **rules**（疎結合の断片。1ファイル＝1トピックで単独ロードでも成立）。

判定: **抜いたら地の文が壊れるか／単独で足し引きできるか**。

#### CLAUDE.md からの @import（第3の置き場）

ファイル分離するが CLAUDE.md 本体の通読文脈に組み込みたいものは、CLAUDE.md 本体に直書きせず `~/.claude/<name>.md` に切り出して `@import` で取り込む。rules ディレクトリの自動ロード対象外で、rules の入場条件は適用しない。

ファイル分離の根拠は複数あり得る（閉じない列挙、新しい根拠が出たら追記）:

- **基盤文脈**: 態度ペルソナ（`persona.md`）・ユーザプロフィール（`user-profile.md`）のように、主役は別ファイルに置き CLAUDE.md には導入の地の文だけを残すと見通しが良いもの。
- **自己言及で揺れる**: 本文書のように CLAUDE.md と rules の境界自体を扱う文書。rules 配下に置くと「rules の入場条件を語りながら自身は入場条件を満たさない」自己矛盾を抱えるため、@import に逃がす。

### rules の入場条件

rules はモデルの事前分布への**残差（誤り訂正項）**であり知識ベースではない。「一般に良いこと」は撃ち込み済みの楔の再入力で、書くと SNR を下げる（自明を書くと裏を読まれる）ので入れない。入れるのは次を満たすもの:

1. **禁止形で書ける＝具体的失敗状態に錨を打てる**（失敗接地の proxy。失敗訂正が支配的多数。ほか「事前分布が割れる所の規約固定（tie-break）」も残差として可）。
2. **その失敗が行動時点で安く自己訂正されない**（モデルがその場で安く回避できるなら常時ロードで先撃ちする限界価値は薄い）。

肯定形でも失敗接地していれば可（例: 「1ツール＝1コマンド」は observed failure 由来）。錨を明記する目的は無矛盾化ではなく、文脈変化時に撃ち直すための**適用条件の自己文書化**。網羅・無矛盾は目指さない——非決定論的対象へのデフォルトであり公理系ではない。例外は前提で、断片はそのために疎結合に保つ。失敗に還元できない良い規範を補う価値は低頻度だが捨てない。

なお本節（および本文書）自身は rules 入場条件を満たさない（規約定義・前提説明であって失敗訂正残差ではない）。だから rules 配下ではなく CLAUDE.md からの @import に置く——「CLAUDE.md からの @import」節の**自己言及で揺れる**根拠に該当する。

### 業務機密は同期される正本に書かない（常時厳守）

業務で得た機密（社内事情・人名・チケット番号・インシデント・面談内容など）を含む規範・知識は、chezmoi で全環境に同期される正本（source）へ流出させない。

- 機密を含む規範は `local-` prefix で `~/.claude/rules/local-*.md` に置く（`.chezmoiignore` で source 非同期、target を直接編集）。
- 機密を、同期される非 local ファイル（`~/.claude/CLAUDE.md`（および import される本節）・paths なし rules・`/knowledge-placement` skill 本体 など）に書かない。auto memory も同様に書かない（私用端末では Obsidian Sync 経由でクラウド同期されうるため）。業務機密は memory ではなく `local-*.md` へ。
- 同テーマで `local-*.md` と同期側が衝突したら `local-*.md` を優先する（より具体的な層だから。自動優先ではなく運用上の取り決め）。
