# コーディングスタイル

コードを書くときの規範。書くべき情報をどの層に残すかと、その層ごとの作法を扱う。

| 層 | 書くもの | 意味 |
|---|---|---|
| コード本体 | **How** | どう実現しているか。識別子と構造で語る。コメントで言い直さない。 |
| テストコード | **What** | 何を保証しているか。テスト名・assert の対象で観測可能な振る舞いを示す。 |
| コミットログ | **Why** | なぜこの変更をしたか。背景・経緯・トレードオフ。 |
| コードコメント | **Why not** | なぜ自然に見える別の選択肢を取らなかったか。コードからは読み取れない判断・制約・落とし穴。 |

## 適用ルール

- **デフォルトはコメントを書かない。** コードコメントは「Why not」が必要な箇所だけに絞る。識別子と構造で How が読めるなら、それを言い直すコメントは削除する。
- **What を語るコメントは禁止。** 「ユーザを取得する」「ループを回す」のような what-it-does 注釈は冗長。読み手はコードを読める。
- **Why（経緯）はコミットログに書く。** 「issue #123 の修正」「X チームの依頼で追加」のような経緯情報はコードに残すと腐る。PR 本文・コミットメッセージに置く。
- **コードコメントに残す「Why not」の典型:**
  - 隠れた制約（「API がこの順序を要求する」「並列実行すると競合する」）
  - 直感に反する選択（「ここでキャッシュしないのは ... のため」「一見冗長だが ... の理由で必要」）
  - 既知のワークアラウンド（「ライブラリの bug #456 を回避するため」）
  - 微妙な不変条件（「呼び出し前に lock 済みであることを前提とする」）
- **テストには What を残す。** テスト名・describe/it ブロック・assertion メッセージで「何の振る舞いを検証しているか」を明示する。実装詳細（How）ではなく観測可能な振る舞いを書く。
- プログラミング言語不問。Python の docstring、Rust の `///`、TS の JSDoc も同じ原則に従う。公開 API の型注釈や引数説明は別物（インターフェース定義）なので対象外。

## 記述言語

- **コードコメントは日本語でも英語でも可。** プロジェクト既存の流儀があればそれに従う。混在していても揃え直す必要はない。
- **コミットメッセージの言語も不問**（日本語で書いてよい。プロジェクトに既存の流儀があればそれに従う）。翻訳ツールが前提の現在は英語統一を強制しない。

## 設計原則（tie-break）

- **YAGNI**: 「いつか使うかも」の interface・generics・コンフィグ化はその「いつか」が来てから。Rule of Three（3 回目で初めて共通化）。
- **DRY は意図の重複除去**: たまたま同形でも意味が違うコードは別々に保つ。
- **識別子で How を語る**: コメントなしで概要が掴める命名（`getUserById` > `getUsr`）。略語は業界標準（API, URL, ID 等）に絞る。
