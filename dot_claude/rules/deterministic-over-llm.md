# 決定論層と LLM の分業

スキル・workflow・agent を組むとき、**確定できる処理を LLM の prompt に委ねない。** 件数集計・モード分岐・規約検証（lint）・証跡の包含照合・フォーマット強制・閾値判定など「正解が一意に決まる処理」は script/workflow のコードで実装し、LLM agent には**本質的に非決定論な判断**（命名・層判定・洞察の発見・自然言語の意味解釈・設計トレードオフ）だけを残す。

- 理由（失敗接地）: 確定処理を prompt に書くと出力が LLM の自己申告に依存し、件数を絞る・状態を畳む・ゲートを骨抜きにする drift が起きる。harvest-pipeline / plan-pipeline は、prompt 層で回していた件数集計・命名ゲート・モード封鎖をコードに移して自己申告依存を断った再編で、この残差の出所。
- 判定: その処理の出力に機械照合できる「正解」があるなら決定論層。出力が判断・発見・解釈で正解が一意でないなら LLM。迷ったら **「LLM が嘘をついても script が検出できる形か」** を問う——検出できない＝確定処理を LLM に渡している。
