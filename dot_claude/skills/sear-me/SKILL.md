# sear-me: 前提を炙り出すフェーズ（plan の前段）

依頼を read-only で最小限だけ調べ、隠れた前提・上流の目的・スコープ境界・成功条件・未決事項を最大 2 ラウンドで炙り出し、`~/workspace/tasks/<slug>/premise.md` に整理して終了するスキル。plan の足場を作るのが役目で、自分で計画は立てない（計画は `plan` フェーズの責務）。

「炙る（sear）」＝表面だけ素早く火を入れる。芯まで焼く `grill-me` の軽量版であり、1 時間規模の尋問はしない。grill のエッセンス（前提・目的・スコープ・成功条件・未決事項を表に出す）は残しつつ、尋問の重さを**ハード上限と構造化選択肢**で封じる。

## 厳守プロトコル

- **`premise.md` を書く以外の Write は行わない。** 出力先は `~/workspace/tasks/<slug>/premise.md` 固定。
- フロントマターで Bash/Edit はツール層ブロック済み。Read/Glob/Grep/Write/Agent/AskUserQuestion のみ使う。
- **最大 2 ラウンド × 1 ラウンド最大 4 問のハード上限**を超えて尋問しない。上限は「上限」であって「ノルマ」ではない（曖昧点ゼロなら 0 問で書いて終えてよい）。
- premise.md を書いたらこのスキルは終了する。自分で plan を立てず、実装にも進まない。

## フロー

### 0. Round 0（read-only・最小調査・無質問）

- 依頼文を整理し、**質問を絞るために必要な最小限の調査だけ**を Read/Glob/Grep で行う。自分で埋められる前提は先に埋め、質問に回すのは「調査でも依頼文でも解消しない、本当に曖昧な軸」だけに絞る。
- **網羅的なファイル洗い出しはしない**（それは plan step2 の責務）。Explore 並列調査は原則使わず、どうしても要るときだけ 1 つに留める。
- **XY problem トリガ判定**: 依頼が**手段指定の形**か（「X を使って」「Y で実装して」など、解き方を名指しした依頼か）をここで判定する。手段指定なら、次の Round 1 で**真の課題（X）を問う設問を必須**にする。

### 1. Round 1（構造化・最大 4 問）

- Round 0 で残った高レバレッジな曖昧点を AskUserQuestion で確認する。
- 選択肢は Round 0 の調査から具体化し、各設問は**推奨案を先頭**に置く。自由記述は最小限に留め、クリックで答えられる構造化選択肢を主体にする。
- **Round 0 で手段指定と判定したら**、「そもそも何を解きたいか／別アプローチの要否」を**必須設問として含める**（条件任せにしない）。
- 曖昧点がゼロなら質問せず Round 2 を飛ばして premise.md を書いてよい。

### 2. Round 2（任意・最大 4 問）

- Round 1 の回答が新たな分岐を開いたときのみ実施する。
- **2 ラウンドでハードストップ**（無限尋問を構造で封じる核）。3 ラウンド目には入らない。残った曖昧さは premise.md の Open questions に書き、plan に引き継ぐ。

### 3. premise.md を Write

- `~/workspace/tasks/<slug>/premise.md` を Write で作成する。スラッグは `~/workspace/CLAUDE.md` の命名規則に従う（既存の plan.md があるディレクトリなら同居させる）。
- 構成は下記「premise.md の構成」に従う。
- 書き終えたら、ユーザに「続けて `/plan` を回すと premise.md を足場に起草する」と伝えて終了する。

## premise.md の構成

plan.md の Context / Approach / Verification に流し込める形にする。**軸の数にも「該当時のみ」を効かせ**、機構だけでなく中身でも軽さを担保する。

### コア軸（必須・4 つ）

- **Request**: 依頼の一文要約（言い直し）。
- **Goal / Done + Why**: 観測可能な完了条件＋**その上流の目的**（なぜこれをやるのか。XY problem 検出の足場）。
- **Scope: In / Out**: やる・やらないの線引き。**Out を最低 1 つは引き出す**（過抽象・YAGNI 予防。plan の YAGNI レビュー観点と接続）。
- **Open questions / Risks**: plan が解くべき未決論点＋確定した決定。**手段指定依頼なら「真の課題への正しい解か（代替アプローチの検討要否）」を必ず含める**（Round 0 トリガと連動）。

### 任意軸（該当時のみ・無ければ省略）

- **Constraints**: 技術・互換・時間・譲れない点。
- **Known context**: Round 0 で質問を絞るのに必要だった調査の足場（パス付き）。**網羅調査ではなく「質問の前提になった範囲」だけ**を書く。plan step2 の網羅調査を侵食しない。

## 設計ガード（軽量さの担保）

- **ハード上限**: 最大 2 ラウンド × 1 ラウンド最大 4 問（AskUserQuestion の上限に整合）。
- **聞く前に埋める**: 固定予算は上限であってノルマではない。曖昧点ゼロなら 0 問で premise を書いて終えてよい（適応的 self-fill）。
- **選択肢主体**: 自由記述でなく構造化選択肢でクリック回答。回答が速く予測可能。
- **2 つの核を対称に必達化**: grill の最も効く ①暗黙の過剰要求の除去（Out を最低 1 つ）と ②XY problem 検出（手段指定なら真の課題を必須質問）を、どちらも条件任せにせず機構で担保する。

## やってはいけないこと

- premise.md 以外を Write する（出力は単一成果物に絞る）。
- 自分で plan を立てる／実装に進む（plan・implement の責務）。
- 2 ラウンドを超えて尋問する（軽量さの核。残りは Open questions に回して plan に引き継ぐ）。
- 調査で埋まる軸を質問に回す（先に Round 0 で埋める）。
- Round 0 で網羅調査をやり込む（plan step2 の責務を侵食しない）。
