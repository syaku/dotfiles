# distill: 作業 → 気づき → 洞察 の蒸留

作業の末に呼ばれ、その作業を知識に変換して vault に固定するスキル。`~/workspace/notes/obsidian/Life`（以下 vault）に対して動く。

**呼び出し文脈は3つあり、最初にどれかを見極める**（素材源と力点が変わる）:

- **(a) Claude 起点** — Claude Code がこのセッションでノートを書く/育てた末に呼ばれる。会話文脈が一級の素材源。通常フロー（日次蒸留）。**推奨モデル: Opus**（洞察(B)検出と捏造抑制が力点でモデル差が最大）。
- **(b) input/ 起点（出所を問わない capture（人/AI）・会話文脈なし）** — `input/` に着地した capture（人の生ダンプ、または会話文脈の無い AI 成果物）を AI が drain する。捕捉と AI の処理は非同期（capture は input/ に足すだけ・pages/ は AI 単一 writer）なので、会話に作業痕跡が無いのは**劣化ではなく正常運用**。`input/` の中身が正規の作業リスト。詳細は「input/ drain モード」。**推奨モデル: Sonnet 都度**（主作業は昇格・整形でモデル差が小さく、誤りは承認ゲート＋後段の日次 Opus 見直しで二重に受ける）。
- **(c) backfill（過去日・期間）** — 「今週分を」等、過去を遡る。洞察(B)に絞る。詳細は「backfill モード」。**推奨モデル: Opus**（(a) と同じく洞察生成が力点）。

**モデル出し分けは `/model` 手動切り替えで行う**（subagent 委譲はしない）。忘れた場合の劣化は非対称: drain を Opus で回すのはコスト増のみで無害、日次を Sonnet で回すと品質が落ちるが承認ゲートで捕まる。この非対称性ゆえに手動運用で許容する。将来拡張（自動化フェーズ）として drain を `model: 'sonnet'` の Agent へ委譲する案があるが今はスコープ外。

失敗接地: 過去に (b) で「会話が無い＝素材が無い」と誤認し、mtime 推定に格下げした上で気づき生成を過保護にスキップした。(b) は会話が無いのが当たり前のユースケースであり、格下げ扱いしない。

このスキルが扱う3つの認知対象は**どれもノード（ノート）**にする。区別は保存形式でなく内容:

- **A. 気づき** — 作業を通じて立ち上がった主観的な学び・判断・教訓・方針（「この設計は失敗した」「こうすべきと分かった」「この罠を踏んだ」）。**名付けられる粒度の独立ノートにして概念ノードへリンクし、`#気づき` タグを付ける**。journal の1行 bullet に溜めない。
- **B. 洞察** — ノード A とノード B を**繋いだ瞬間**に初めて見える第三の知見。内容が「関係そのもの」のノート。これが本体の payoff であり、`#洞察` タグで書き出す。
- **T. タスク** — 未着手の行動（① 明示 TODO / ② 次タスク候補 / ③ ノート分析で出た課題）。1 ノート 1 タスクで `pages/` 直下に固定し、`#タスク` タグ ＋ `progress: backlog | ready | doing | done` の 4 値で識別する。**タスクは洞察(B)の素材に含めない**——層が違う（未着手の行動から直接洞察を立てると「弱い繋がりからの捏造」既存規律と摩擦が出る）。代わりに、タスクが生まれた**背景（本文 wikilink で繋いだ元ノート）を洞察素材として扱う**（ステップ3の素材は元ノート側を見る）。タスク単体しか手がかりが無い場合に洞察化したくなったら、**まず気づきノートを起こしてから**それを洞察素材に回す（タスク → 気づき → 洞察 経由必須・直接ジャンプを禁止）。

**3 層の関係**: B（洞察）は A（気づき）の上位構造——気づきを束ねて洞察を作る（既存洞察を束ねたメタ洞察もありうる）。T（タスク）は A/B とは別軸の**行動層**であって、A/B の認知高度には乗らない。よって**1 ノートが複数層を兼ねることはありえない**——既存気づきノートに `#タスク` を後付けして二重タグ化したり、気づきから洞察に「化けさせ」たりはしない（化けるときは新規ノートを起こす。気づきと洞察が同じノートに同居しないのと同じ規律）。

**気づき(A) と「事実・調査結果」を取り違えない（`#気づき` タグの肝）。** 検証で確定した客観的事実・仕様・スペック（例: 「Dataview は ISO 8601 を date 型として解釈する」「Templater 変更は既存ノートに遡及しない」のような**外部検証可能な事実**）は気づきではない。これらは調査記録・事実ノードであり、`#気づき` を付けず**トピックタグだけ**にする。**作業メモ・作業レポートの類い**（「〜作業メモ」「〜スキル新設」「〜検証ログ」等、やったこと／調べたことの記述が主のノート）も、それ単体では事実記述なので気づきではない＝付けない。判別: 気づきは「作業を通じて立ち上がった主観的な学び・判断・方針」、事実は「外部で検証できる客観」。層の高度は **事実/仕様 → 気づき → 原理・方針 → 洞察** の順に上がり（後述「タイトルの付け方」）、`#気づき` が乗るのは中2層（気づき・原理・方針）、`#洞察` は最上層、事実/仕様・作業メモは無印。

**決め手はタイトルの高度であって、AI Context の「検証済みファクト」自称ではない**（自称は確認手段の話で分類の決め手にならない。失敗接地: この自称に引きずって誤判定した）。特定ツール名を冠した**狭いスペック**（「Dataview は ISO 8601 を date 型で解釈する」「Templater 変更は遡及しない」）は事実だが、主語をツールから一般化して「踏んだら当てにできない」という**教訓に上げたタイトル**（「ファイル同期はタイムスタンプを保持しない場合がある」）は、検証で確認した事実が土台でも**気づき**（スキル「主語を一般化すると気づきが洞察になる」と同じ昇格——一般化で1層上がる）。判別の問い: **特定ツールの狭いスペックか／一般化された踏んで学んだ教訓か。**迷ったら：それは「踏んで学んだ判断」か「外部で検証できる狭い事実」か。

失敗接地: 旧運用は A を journal の1行 bullet に留めた。Obsidian は Logseq と違い要素単位リンクが無く、グラフ市民権の単位が file なので、bullet は backlink に現れず関係に参加できない＝死文化する。素材がグラフに残らないので distill は毎回会話からゼロ構築するしかなく、既にグラフにある素材を活かせない。気づきもノード化し、グラフ上に素材を残す。

## 厳守プロトコル

- **捏造補完しない。** 元になった作業・ノートから「復元できる範囲」に留める。繋がりが弱いのに無理に洞察をでっち上げない（vault 既存規約「元メモが断片的でも無理に補完しない」を踏襲）。
- **ノート作成（気づき(A)・洞察(B) とも）と既存ノートの実体改変は、提示して承認を得てから。** ただし気づきノードは per-note でなく**トリアージ一覧**（各気づき＝名付けて新規ノード化、関連既存ノードへのリンク先、タイトル案）を一括提示して承認する（軽量化）。**journal の `### 気づき・洞察`・`### 今日触れたノート` は Dataview 自動集約に移行済み**——distill が手でリンク列を維持する必要はない。2欄は別基準で答える問いが違う: `### 今日触れたノート` は末尾 `## 更新履歴` の当日 `[[日付]]` 実リンクを被リンク集約した「**触れた日**」基準の活動ログ（全行に行頭アイコン: 💎洞察／💡気づき／📄その他＝作業レポート・メモ等。無印行を作らず表示の頭を揃える）、`### 気づき・洞察` は frontmatter `createdAt` 基準で「**得た日（ノード誕生日）**」に載る年代記。よって新規に名付けた気づき／洞察ノート（createdAt=当日）はその日の年代記に出る。既存ノードへ畳んだ再認識（createdAt は過去）は年代記には出ず、活動ログに 💡／💎 付きで出るだけ——その知見は誕生日の年代記に既出なので二重掲載しない設計。
- **pages/ ノートを作成・改変したら frontmatter `updatedAt` を実行時刻の ISO-T 形式（`YYYY-MM-DDTHH:mm`）で必ず打つ。** `updatedAt` は recency 索引（`MOC` 等の現状表示）。filesystem 時刻（`file.mday` 等）は Obsidian Sync で同期時刻へ飛ぶので使わない。`createdAt`・`updatedAt` とも ISO-T 形式（`YYYY-MM-DDTHH:mm`）で書く（Dataview は ISO-T のみ日付認識）。
- **pages/ ノートを作成・改変したら、末尾 `## 更新履歴` に `- [[当日日付]] — 修正理由` を冪等追記する（当日分が既にあれば足さない）。** これが daily「今日触れたノート」の被リンク集約（`FROM [[日付]]`）の供給源＝**日次記録の正本**（原則: 記録＝実リンク（不変）／索引＝Dataview）。理由は粗くてよい。**journal 日付への wikilink は `## 更新履歴` に集約**し本文に散らさない（本文の `→ [[日付]]` 相互参照もここへ寄せる）＝`FROM` が編集日と厳密一致する。決定論的・網羅的な履歴はバージョン管理の領分で本運用は担わない。
- **外部インポートを触らない**: `imports/kindle/`・`imports/wallabag/` は読み取りのみ。リンク先に使うのは可だが編集・リネームは禁止。
- **vault のタグ規約に従う**: 1ノート3タグ程度、大枠のカテゴリのみ、日本語優先、既存タグ再利用。**気づきノートには必ず `気づき` タグ、洞察ノートには必ず `洞察` タグ、タスクノートには必ず `タスク` タグを含める。事実・調査記録には付けない**（前掲の A／事実 区別。トピックタグのみ）。
- **タスクの `progress:` フィルタを daily の掲載クエリ（`### 今日触れたノート`・`### 今日生まれたタスク`）に混ぜない**。誕生日記録の不変性を保つ（done になっても誕生日 daily には残る）。`MOC/タスク.md` だけが `progress` で 4 段（doing/ready/backlog/直近 1 ヶ月の done）に切る現状表示を担う。
- **タスクは `pages/` 直下に置く・専用フォルダに隔離しない**（過去 `pages/projects/` のフォルダ失敗を繰り返さない。束ねるのは `MOC/タスク.md` ＋ `#タスク` タグ）。
- **タスクから直接洞察を立てない・気づき経由必須**（冒頭 T の規律）。タスクは「未着手の行動」で層が違うので洞察素材に含めない。洞察化したい時はまず気づきノートを起こしてからそれを素材に回す。
- **タスクノートの `status:` を `done` に書き換えない**（タスクの完了は `progress: done` で表す。`status:` はノートライフサイクル＝`active` のままで触らない。`status:` と `progress:` は意味直交）。
- **抽出時のラベル ①②③ を作成後のタスクノートに残さない**（frontmatter にも本文にも `①` `②` `③` 文字を書かない）。出自は本文 wikilink で繋いだ元ノート ＋ `## 元ノート(なぜ重要)` の有無で実質的に区別可能。
- **タスクノート本文の `## やること` を `- [ ]` チェックボックスで書かない**（plain な `- ` 箇条書き。Obsidian Tasks プラグインのインデックスとは二重化させない）。

## タイトルの付け方（気づき・洞察の命名）

ノードの価値は graph に出るタイトルが握る。長く具体ディテールを積んだタイトルは「何が学びか」を埋もれさせ、抽象に振りすぎたタイトルは要点を蒸発させる。軸は「具体↔抽象」ではなく **要点を plain に外さず言い切れているか**。以下は実セッションの改名作業（2026-06-06）と命名の複雑化（2026-06-07）からの失敗接地。

- **1 タイトル＝1 要点。条件と結果を二重に重ねた複文・要素を二つ詰めた長文にしない。** 「整合を後追いの横断パスに切り出すと責務が二箇所に割れる」✗→「整合を後から足すと責務が割れる」○、「手段で指定された依頼を額面で受けると枠組みが崩れる」✗→「手段で来た依頼は真の課題を隠す」○。洞察も同じで、硬い比喩動詞より日常語にする（「概念は軸に割ると例外が見える」✗→「別物を同じ名前で呼ぶと噛み合わない」○）。問題は長さ自体でなく**一文に二つ詰める複雑さ**——既存の良いタイトルは一要点の短い言い切り。失敗接地: 2026-06-06 に続き 2026-06-07 も複文で長く複雑化を繰り返し、毎回ユーザ指摘で短縮した。**初稿のタイトルが複文（「〜すると」「〜を〜と〜」）になっていたら、提示前に一要点へ割る。この複文チェックは散文の戒めに留めず、ステップ4の決定論 grep ゲート＋要点監査で機械的に効かせる**（提示前チェックを散文に置いたため 2026-06-09 も再発したので、○×の自己採点でなく必須の機械ゲートに変換した）。
- **解・抽象的な徳でなく、避けたい失敗・問題をタイトルにする。** 「単一オーナー化が解」✗→「順序の決まらない同期はどの基盤でも正統を奪い合う」○。「誠実さは件数ノルマに優先」✗→「数合わせのために中身を水増ししない」○。解や徳は本文へ逃がす。
- **plain な確立語を使い、造語・狭い実装語を避ける。** `対の系統`（造語）・`mtime`（更新日時しか指さない狭語）・`ISO-T`（非正式）・`未文書バグ`（非語）✗→`ペア`・`タイムスタンプ`・`ISO 8601`・`未報告のバグ`○。語の存在・正式さに迷ったら一次情報で確認する（自分の記憶を正としない）。
- **scope は固有名詞で狭めず hedge で合わせる。** 一般現象なら一般語に上げ「場合がある」等で正直に留める（`Obsidian Sync は…mtime を…`→`ファイル同期はタイムスタンプを保持しない場合がある`）。**ただしツール固有のクセ・仕様は固有名詞を残す**（Dataview の日付解釈・Templater の遡及なしは一般現象でないので名前を残す）。判別の問い: 一般現象か、そのツール固有か。
- **語そのものに nuance を運ばせる。** `食い違う`（単にズレる）と`食い合う`（正統を奪い合う）は別物。要点の nuance を持つ語を選ぶ。
- **抽象は本文/リンクが事例で接地している時だけ許す。** 主語を上げると気づき→洞察に化ける（`Obsidian は片方を直すと…`＝気づき → `ペアの片方を直すともう片方が古いまま残る`＝Obsidian 接地が外れた独立洞察）。接地のない抽象は A を B に水増しする空語。
- **層で最適な高度が違う。一律に短く/抽象にしない。** 作業ログ・調査記録（`wezterm の IME ズレと nightly 移行` 等）は具体のまま据え置く＝抽象化すると実害（何を調べたかが消える）。事実/仕様 → 気づき → 原理・方針 → 洞察 の順に高度が上がる。
- **読み手はソフトウェアエンジニア。気づき/洞察の「なぜ重要」「応用・次アクション」は、ソフトウェア開発に転用できる形を最低1つ接地させる**（トピックをソフト開発に限定する話ではなく見せ方の話）。例: 同期・PKM・vault 運用の知見でも、デバッグ／依存管理／コード生成／設定移植／可観測性の語彙で「次の開発でどう効くか」を書く。背景: 利用者本人が SWE で、転用可能な書き方だと活用率が上がるとの本人申告（2026-06-07）。
- **タスクのタイトルは別軸——動詞主体の短句**（`〜する`／`〜化する`／`〜を確認する` の体言止め／動詞主体）で何の行動かを言い切る。気づき/洞察の「避けたい失敗を名指す」規約とは別軸。複文 grep ゲート（`、|すると|したら|つつ|（|\(`）はタスクにも同じく適用する（複文化予防は決定論ゲートで全タイプ共通）。

## フロー

### 1. このセッションで触れたノートを把握

- まず会話の文脈から、このセッションで作成・編集したノート（vault 内）を洗い出す。
- 補助として mtime も使える: `ls -lt ~/workspace/notes/obsidian/Life/pages/*.md` 等で直近更新を確認。ただし**会話で実際に触れたものを正**とし、mtime だけで無関係なノートを巻き込まない（失敗接地: Obsidian Sync で他端末由来の更新が mtime を新しくすることがある）。
- **会話に作業痕跡が無い場合**（人が input/ に生ダンプして後から呼んだ (b) のケース）は、ここで mtime 推定に格下げしない。`input/` の中身（`README.md` 除く）を正規の作業リストとして扱う「input/ drain モード」へ進む。
- **(a) 日次蒸留では、カーソル時刻より新しい `createdAt` を持つ pages も対象に含める**（前回日次蒸留以降に drain で昇格したノートを拾う）。drain は別セッション・別日に走り会話文脈に残らないので、これを拾わないと Sonnet が確定した命名・層判定ミスを Opus が見直さず固定する。カーソルは訂正ログ `pages/distill運用ログ.md` の**本文先頭（frontmatter `---` の外）の plain text 行** `最終日次蒸留: YYYY-MM-DDTHH:mm`（ISO-T テキスト）＝1値だけ持つ状態でノートの frontmatter とは別物。**比較するのは、その1値と各 pages の frontmatter `createdAt`**（filesystem 時刻でなく frontmatter。AI が ISO-T テキストで打つ不変値。Obsidian Sync が飛ばすのは filesystem mtime/birthtime のみでテキストは飛ばない既存規律に乗る＝`distilled` 等の永続フラグは導入しない）。抽出は既存の `FROM "pages" WHERE ... createdAt ...` 構文を踏襲するか Grep で frontmatter を当たる。
  - 冪等性のトレードオフ: 一度日次に通したが洞察が出ず保留にしたノートは、カーソル更新後は拾われなくなる（保留は完了報告に残す既存の仕組みで拾う）。観測フェーズは小件数で再浮上コストが小さく許容する。
  - 初期カーソルは器ノート新設時刻に置く。**それ以前に作られた既存 pages はカーソルより古く、日次の対象選定には上がらない**（過去 pages 全量を初回に巻き込むのを避ける意図的な設計）。過去の未蒸留ノードを拾い直したいときは backfill モードで明示的に対象期間を指定する。

### 2. 気づき(A)をノード化する（既存ノードと突き合わせ → 畳むか新規か）

- 当日ファイル `~/workspace/notes/obsidian/Life/journals/<YYYY-MM-DD>.md` を特定する（日付は会話の現在日）。`### 今日触れたノート`（被リンク集約＝触れた日）・`### 気づき・洞察`（`createdAt` 絞り＝得た日）はいずれも Dataview で自動表示するので手で列挙・維持しない。
- 気づきを拾う。**(a) 会話から**／**(b) 手書きメモ起点は同席ユーザに聞いて**（無い会話から想像で復元しない）。本文に「検証で確認」等と明記された事実はそのまま転記してよい。
- **関連既存ノードと突き合わせてリンクする（最重要）。** 各気づきに関連する既存ノート（その日の作業ノート・概念ノート・既存洞察）を Grep/Read で探し、**リンクを張る**。目的は重複排除でなくリンク——失敗は重複でなく orphan（リンク欠如）の方で、**重複生成はその主題が繰り返すシグナルの強さ**。明白に同一物の既出だけ既存ノードに畳み、**迷ったら分けて作りリンクする**（畳んで捕捉・リンクを落とす方が高くつく）。
- **新規の気づきは名付けてノード化する。** 単位は**名付けられる粒度**——名詞タイトルだけのノートでも、ノードとして関係を集めるなら有効。名付けられないほど小さい断片は名付けられる粒度まで束ねてから（段落ごとに割らない）。命題化できなくてよい＝名付けられれば足りる。内容と関係の両方に気を配り、本文から概念ノードへ wikilink を張る（グラフに出るのは frontmatter link と本文 wikilink）。frontmatter `tags:` に **`気づき` を付ける**——ただしその内容が検証済みの事実・調査結果なら気づきではない＝付けない（前掲の A／事実 区別）。**作成・改変したノートには末尾 `## 更新履歴` に当日 `[[日付]]` リンクを冪等追記する**（daily 被リンク集約の供給源＝日次記録の正本）。
- **作成後、関連既存ノードの側からも新規気づきへ wikilink を張り返す（双方向に辿れるように）。** 既存ノートへの追記は最小限・本文の自然な位置に。
- **journal の `### 気づき・洞察`／`### 今日触れたノート` は手で維持しない**（Dataview が自動集約: 今日触れたノートは被リンク＝触れた日、気づき・洞察は `createdAt`＝得た日で絞る）。distill の役目は各気づきノートに `#気づき`／`#洞察` タグ・正しい `createdAt`（ISO-T）・末尾 `## 更新履歴` の当日 `[[日付]]` リンクを担保すること＝それで daily に自動掲載される。journal stream に旧来の手書き prose が残っていれば、**その知識が行き先ノートに在ることを確認の上で**撤去し、TODO・予定・ephemera だけ残す（消す前に行き先を確認＝情報ロス防止）。当日ファイルが無ければ最小見出しだけ作る。
- 以上を**トリアージ一覧として提示→承認→作成/リンク化**（厳守プロトコル参照）。
- friction 注記: ノード化は bullet より重く、重いと『書かなくなる』で捕捉が落ちる。低摩擦に留める（テンプレ流用・最小 frontmatter）。

### 2.5. タスク(T)を抽出する（3 種ラベル → 承認 → ノート化）

**この節は気づき(A) と洞察(B) の間に挟まる**——タスクは未着手の行動で層が違うので、洞察素材としては扱わない（タスクが生まれた背景の元ノートを洞察素材に回す。冒頭 T の規律を踏襲）。

#### 抽出対象 3 種（ラベル ①②③ は抽出時の制御用・作成後のタスクノートには残さない）

- **① 明示 TODO** — 元ノートに「TODO」「未実施」「やる」等の語が plain にある未着手記述。
- **② 次タスク候補** — 元ノート文面に「次は〜」「後で〜」等の先送り表明がある。
- **③ ノート分析で出た課題** — AI が文面の論理ギャップ・矛盾・未解決を分析で抽出。**必ず「なぜ重要」を添える**＝承認ゲートで弾けるようにする。

**ラベル ①②③ は frontmatter にも本文にも残さない**（出自は本文 wikilink で繋いだ元ノート + H2 `## 元ノート(なぜ重要)` の有無で実質的に区別可能）。気づき/洞察も由来分類を構造化キーで持たないのと同型。

#### 抽出の流れ

1. **元ノートをスキャン**（このセッションで触れた pages + input 中身。後述「3 モード差分」で対象範囲が変わる）。
2. **候補抽出**（3 種ラベル付き）。③ には「なぜ重要」を添える。
3. **既存タスクノートと突き合わせ**: `WHERE contains(tags, "タスク")` のクエリ or `rg -l '#タスク'` 相当で既存タスクを洗い、明白に同一の TODO は既存タスクへリンクで畳む（突き合わせのキーはタイトル・内容であってラベル ①②③ ではない）。**迷ったら分けて作りリンクする**（気づきと同じ規律 = 失敗は重複でなく orphan）。
4. **承認ゲートのトリアージ一覧**（タスクごと）:
   - タイトル案（動詞 + 目的語の短句。「タイトルの付け方」節の grep ゲートを通す）
   - 元ノート wikilink
   - ラベル（① / ② / ③）← 抽出時の制御用に提示
   - **初期 `progress`**: 全件 `backlog` 固定（`ready`/`doing` への昇格は人の判断）
   - なぜ重要（③ 必須・① ② 任意）
5. **承認されたものだけ作成**。`pages/` 直下に書き、frontmatter は実値で書く:
   - `tags: [タスク, ...]`（大枠カテゴリ 1〜2 ＋ `タスク` の計 3 タグ程度）
   - `progress: backlog`
   - `createdAt:` / `updatedAt:` を実行時刻の ISO-T 形式（`YYYY-MM-DDTHH:mm`）で
   - `templates/pages.md` 由来の `status: active` / `aliases: []` は触らずそのまま残す（`progress:` と意味直交）
   - **元ノートは本文中の wikilink で繋ぐ**——気づきノートと同型の規律で、frontmatter `source:` キーは使わない（`source:` の構造化は洞察ノートが Dataview 抽出の都合で必要なだけで、タスクは本質的にリンクなので本文 wikilink で足りる）
6. **逆リンク**: 元ノート側にもタスクへ wikilink を張り返す（気づきノードと同型の双方向リンク規律）。
7. **`MOC/タスク.md` への手動追記は不要**（Dataview が自動集約）。

#### タスクノートの本文の最小構造

vault CLAUDE.md の callout 規約に準拠した H2 最大 3 つ。

- `> [!NOTE] AI Context` callout に**主題・用途**を 1 文（例: 「提案書のレビューと修正提出」「support 顧客連絡先 DB のクリーンアップ」）。**「なぜ立てたか」「なぜ重要」は callout に書かない**（下の H2 `## 元ノート(なぜ重要)` と分担）。
- `## やること` — 観測可能な完了条件を **plain な箇条書き（`- `）** で列挙。**`- [ ]` チェックボックスは使わない**（Obsidian Tasks プラグインのインデックスに乗ると二重管理になるため。Tasks プラグインの中央集約方式は採用しない）。
- `## 元ノート(なぜ重要)` — ③ の場合は必須で「なぜ重要」を本文展開。① ② は省略可（本文中の wikilink で繋いだ元ノートを辿れば足りる）。
- `## 更新履歴` — 既存規律と同じく当日 `[[日付]]` 実リンクを冪等追記。

**専用テンプレ `templates/task.md` は作らない**（Templater は `pages/` 直下を `templates/pages.md` で自動適用するので task 専用テンプレを起動できない・AI が実値で書き込む前提・YAGNI）。

#### progress 4 値の定義

| 値 | 意味 |
|---|---|
| `backlog` | 未着手・いつかやる（next-up に上げる前のプール） |
| `ready` | 着手可能・次に拾うつもり（即実行できる状態に揃っている） |
| `doing` | 進行中・現在の WIP（実際に手を付けている） |
| `done` | 完了 |

標準フローは `backlog → ready → doing → done`。途中スキップ（例: `backlog → doing`）も許容（着手のリズム次第）。

#### progress の遷移経路（誰が・どこで）

- **主流路**: **`progress: done` への遷移は distill が AI 経由で行う**。作業末に distill が呼ばれた際、当該セッションで生まれた作業レポート（input/ ノート or 日次の会話痕跡）を元に「このタスクは完了した」を検出し、承認ゲートで「done 化候補」として提示 → 承認後 AI が `progress: done` ＋ `updatedAt` を更新。`## 更新履歴` に当日 `[[日付]]` ＋「完了」を追記。
- **サブ路**: `backlog → ready → doing` 遷移は**人が手で frontmatter `progress:` を編集する**（pages/ 単一 writer の例外。`progress` キーの値変更のみ人が触ってよい——タスクの「着手する」「次に拾う」は人の意思決定）。書き換え時に `updatedAt` も同時に更新する規約。`status:` は触らない（ノートライフサイクルキーは Templater 自動付加のままに残す）。
- **手動 done 化も例外として許容**（人が直接 `progress: done` に書き換える）。ただし基本は distill 経由の AI 書き換えを推奨経路とする。

### 3. 繋がりから洞察(B)を検出

- このセッションのノートが、**既存ノートのどれと繋がるか**を探す。Grep/Read で関連語・既存タグ・既存トピックノートを当たる。
- 「個別には既知だが、繋ぐと第三の知見が出る」ものだけを洞察候補にする。単一観測・単一ノートの感想は A（気づきノード）であって B ではない。
- 「先週/過去の洞察と同じ筋」を見つけたら、それ自体が再発パターンの洞察になりうる（`MOC/洞察.md` や既存 `#洞察` ノートを参照）。

### 4. 洞察ノートを提示 → 承認 → 作成

- 候補ごとに次を**この順で**提示する（タイトルを先に書かない＝claim-first）:
  1. **繋いだ元ノート群**
  2. **主張**: 見えた洞察を一文で言い切る（複文OK＝本文「見えた洞察」と同じ。ここは漏れ無しでよい）
  3. **タイトル**: 主張から**述語を1つ選び、条件節（「〜すると」「〜を…と」等）を捨てて**圧縮する（命題そのものを詰めない。「圧縮」だけだと複文シルエットを温存するので捨てる対象を明示する）
  4. **機械ゲート（決定論・必須）**: 候補タイトルを `printf '%s' "<title>" | grep -nE '、|すると|したら|つつ|（|\('` に通し、**結果を転記**する（ヒット箇所／0件）。ヒット＝複文疑い → 3 に戻り再圧縮。○×の自己申告でなく**機械が弾いた事実**を出す。grep は表層の典型複文（読点・条件接続・並列「つつ」・括弧）を決定論で捕まえる主経路で完全網羅ではない＝意味レベルは 5 に委ねる。字数は複文の proxy として弱い（一要点でも日本語は20字超が普通）ので、このゲートに混ぜず命名の soft な目安（本節冒頭「長く具体を積むと要点が埋もれる」）に留める
  5. **要点監査（意味レベル・割る作業を強制）**: 「このタイトルを2つの主張に割るとしたら何と何か。割れないなら『割れない』と明記」。Yes/No の自己申告でなく**割る作業の痕跡**を提示に出すことで、表層 grep が拾えない意味レベルの並列2要点（読点も接続詞も無い）を露見させる（grep が「結果の転記」で自己採点の甘さを塞いだのと同じ機序を意味レベルにも効かせる）
  6. **なぜ重要**（1行）
  - 失敗接地: 洞察タイトルは「主張をタイトルに詰める」conflation で複文化し再発した（2026-06-06/07/09）。決定論 grep を主経路に、意味レベルだけ人レビューに分けることで silent な自己採点の甘さを塞ぐ。
- 承認されたものだけ `pages/` に作成する。形は `templates/insight.md`（洞察テンプレ）に従うが、Claude Code から書くので Templater 構文は使わず**実値で埋める**:
  - `createdAt:`・`updatedAt:` とも現在時刻を ISO-T 形式（`YYYY-MM-DDTHH:mm`）で（Dataview が日付認識する機械可読）
  - `tags:` に `洞察` を含める（＋大枠カテゴリ1〜2個まで）
  - `source:` に繋いだ元ノートを Obsidian プロパティ形式の wikilink リストで列挙: 各行 `  - "[[ノート名]]"`
  - 本文の AI Context callout・見出し（見えた洞察／なぜ重要／応用・次アクション）を埋める
- 作成後、**繋いだ元ノートの側からも**新しい洞察ノートへ wikilink を張り返す（双方向に辿れるように）。既存ノートへの追記は最小限・本文の自然な位置に。
- `MOC/洞察.md` は `#洞察` を Dataview で自動集約するので**手動追記は不要**。手で束ねたいテーマがある場合のみ MOC 本文にリンクを足す。

### 5. 完了報告

- 作成した気づきノード／既存へ畳んだ件数・作成した洞察ノート（パスとタイトル）・**作成したタスクノート（パスとタイトル・初期 `progress: backlog`）**・張ったリンク・各ノート `## 更新履歴` への当日リンク追記を箇条書きで報告する。
- 承認を保留した候補があれば、その理由（繋がりが弱い等）を添えて残す。
- **done 化候補の検出と提示**: このセッションの作業レポート（input/ ノート or 会話痕跡）から、既存タスクノート（`#タスク` ＋ `progress != "done"`）の完了を読み取る。検出ロジックは「作業レポート中で当該タスクの `## やること` が満たされた」「明示的に『〜は完了した』との記述がある」等の証拠ベース（推測で done 化しない）。候補があれば**承認ゲートで「done 化候補」として提示**: タスクのリンク・現 `progress:` 値・完了根拠（作業レポートのどの記述か）。承認されたものだけ `progress: done` + `updatedAt` を実行時刻 ISO-T で更新し、`## 更新履歴` に当日 `[[日付]]` ＋「完了」を冪等追記する。`status:` は触らない。done 化候補が無ければ何もしない（無理に done 化しない＝捏造補完しない既存規律と同じ）。
- **(a) 日次蒸留では、運用ログ `pages/distill運用ログ.md` へ訂正サマリを追記する。** 承認ループで人が直した差分を観測データとして残す（自動化フェーズの教師データ）。記録粒度（小件数前提で全件・最小テンプレに留め friction を避ける）: 日付／モード(drain|daily)／起動モデル／タイトル訂正 `元→修正後` 全件ペア／層判定訂正は4層遷移（事実・仕様→気づき→原理・方針→洞察）で／洞察却下の件数＋理由。**各訂正には対象ノートの出自を付す**——カーソル経由で拾った＝drain(Sonnet)産の見直しか、当日会話産＝Opus 初出か。日次は両方を「モード daily・Opus」で処理するので、出自を付さないと「Sonnet が何を外したか」が「Opus が何を直したか」に埋もれ、教師データの肝（drain=Sonnet の癖）が遡れなくなる。「起動モデル」欄が drain=Opus・daily=Sonnet なら `/model` 手動運用の取りこぼしシグナル。承認ループの副産物として取り、新規の重い記録作業にしない。
- **(a) 日次蒸留の完了時に、上の運用ログ追記とセットでカーソル `最終日次蒸留:` を実行時刻（ISO-T）へ更新する。** これで次回はこの時刻より新しい `createdAt` の pages だけが対象に入る（ステップ1の対象選定と対）。運用ログ追記とカーソル更新は同じ完了動作の対——片方だけやって片方を落とすと観測と対象選定がズレるので、必ず一緒に行う。

## input/ drain モード（出所を問わない capture（人/AI）を処理）

`input/` に着地した capture（人の生ダンプ、または会話文脈の無い AI 成果物）を AI が処理するケース (b)。会話に作業痕跡が無いが**対象は当日の捕捉**で、捕捉と AI の処理は非同期（capture は input/ に足すだけ・pages/ は AI 単一 writer）。backfill（過去日）とは別物。力点（気づき(A)も洞察(B)も両方やる）は (a) と同じ。

**推奨モデル: Sonnet 都度。** 主作業は input→pages 昇格・整形でモデル差が小さく、命名・層判定の誤りは承認ゲート＋後段の日次 Opus 見直し（カーソルで drain 産ノートを拾い直す）で二重に受ける。drain では運用ログへの記録もカーソル更新もしない（日次蒸留が担当する）。

### 通常フローとの差分
- **`input/` の中身が正規の作業リスト。** `input/*.md`（`README.md` は除く）を読む。出所が人でも会話外 AI でも capture は既に済んでいる（input/ に書き下ろされている）ので、(a) のように会話から拾う必要も、想像で復元する必要もない。曖昧な点だけ同席ユーザに確認する。
- **処理前スナップショット（消失検知の基準）。** 読込直後に `ls ~/workspace/notes/obsidian/Life/input/*.md | sort` の出力をそのまま控える。処理途中で取り直さない（照合の基準を不変にする＝長い昇格処理の context rot で劣化させない）。
- **input ノートを pages/ ノードへ昇格させる。** 名付けられる粒度で気づき(A)ノード化（通常フローのステップ2と同じ規律: 既存ノードと突き合わせ→関連へリンク、迷ったら畳まず分けて作る）。昇格時に frontmatter `createdAt`・`updatedAt` を ISO-T 形式（`YYYY-MM-DDTHH:mm`）の実値で書く。1つの input から複数ノードが出てよい。
- **昇格で input 名が変わる／複数に分割される場合、元 input を指していた既存 wikilink を昇格先へ張り替える**（複数分割なら主たる行き先へ）。被リンクの所在は想像で決めず `rg -l '\[\[<元 input 名>\]\]' ~/workspace/notes/obsidian/Life` で機械的に洗う（スコープは名が変わった input に限る＝毎 drain 全件 grep ではない）。0 ヒットなら張り替え不要で即終了。同名昇格を既定前提にしない（`alwaysUpdateLinks` はリネーム時のみ作用し、mv→新規 pages 作成では効かない＝自動では張り替わらない）。
- **処理済みの input 原本を `archive/input/` へ移動する。** ただし **mv の前に `ls ~/workspace/notes/obsidian/Life/input/*.md | sort` を再実行し、処理前スナップショットと集合差分（`comm` 等）を取る。欠落があれば archive mv を実行せず、欠落ファイル名を報告してユーザ判断を仰ぐ**（原本ロストの上に mv で状態を複雑化させない。昇格済みノートは残す）。差分なしなら `mkdir -p ~/workspace/notes/obsidian/Life/archive/input` の上で `mv`。消さずに退避し provenance を残す＝inbox を空に保ち、残量＝未処理キューの可視化を維持する。drain 中は Obsidian Sync 競合で input 原本が消えうるので、**同席ユーザがいれば他端末の Obsidian を閉じるよう一言促す**（ユーザへの依頼＝副次的防御。主防御は上の照合・停止）。
- **洞察(B) は通常フロー（ステップ3〜4）どおり。** 昇格したノードと既存ノートの繋がりを探し、候補を提示→承認→作成→逆リンク。
- **タスク抽出の差分**: ステップ 2.5 の 3 種（① 明示 TODO / ② 次タスク候補 / ③ ノート分析で出た課題）すべてを抽出する。input/ の生ダンプは TODO・課題が多いので主要な発生源。承認ゲートのトリアージ一覧で人が採否を決め、③ は「なぜ重要」必須。抽出元の input ノートが pages へ昇格する場合、生まれたタスクの**本文 wikilink** はその昇格先 pages を指す（input/ のパスを残さない）。input が昇格せず純粋に TODO リストだった場合は、本文 wikilink を張らない（タスク単体で立ち、`## やること` だけで足りる）。

### やはり出ないなら出さない
- input が空、または所感なしの断片だけで繋ぐ素材が無いなら、`A 止まりですらない`と正直に報告し、洞察をでっち上げない（backfill の同名節と同じ規律）。原本の archive 退避だけ行う。

## backfill モード（過去日付の遡り distill）

「今週分を distill」「先週を振り返って」など**過去の日付・期間**を対象に起動された場合のモード。通常フローとの違いは入力源と力点。**推奨モデル: Opus**（(a) と同じく洞察(B)生成が力点でモデル差が最大）。

### 通常フローとの差分
- **力点は洞察(B)。気づき(A)は基本やらない。** 過去日の「その場の気づき」は会話文脈が無く、書けば捏造になる。journal に書かれていない感覚を想像で埋めない。よってステップ2（気づきのノード化）は原則スキップする。
- **過去 journal を埋めない。** 空の作業メモを想像で補完するのは禁止。その日の Dataview「今日触れたノート」は当時のままでよい。

### 素材の集め方
- 対象期間の作成ノートは **frontmatter `createdAt` を正**にする。**filesystem の birthtime は使わない**（失敗接地: ノートを別フォルダへ mv すると birthtime が移動日にずれ、本来の作成日と乖離する。実際に sub_agents の救出移動で 2025-12-22 作成のノートが移動日付に化けた）。例: `head -3 pages/*.md` で frontmatter 日付を拾う。
- その日の journal `## 作業メモ` に手書きの記述があれば素材に加える（空の日も多い）。

### 出ないなら出さない（最重要）
- 中身が空のスタブ群（97B 前後の空フレーム等）は繋いでも洞察にならない。**「A 止まりですらない」と正直に報告**し、無理に B をでっち上げない。
- 既に ADR・MOC・memory で分厚く自己文書化済みのクラスタ（例: homelab/Proxmox）は、繋いでも**既出の焼き直し**になりがち。新規の創発が無ければ作らない。
- ただし**空の器・放置ノートが複数あること自体**は、繋ぐと再発パターンの洞察(B)になりうる（例: 空スタブ＋空 journal 作業メモ＋死んだフォルダ＝「器だけ作って放置する」failure mode）。空虚さもパターンの素材。

### 出力
- ステップ4と同じ（洞察ノートを提示→承認→作成→逆リンク）。`createdAt` は実行時刻（過去日でなく今日）。AI Context に「過去 distill で見えた洞察」である旨を記す。

### タスク抽出の差分（3 重ゲートで ② ③ を機械的に封鎖）

backfill は過去日の感覚を想像で埋める捏造リスクが高いので、タスク抽出も限定する。**① 明示 TODO のみ抽出**し、② と ③ は機械的に封鎖する。3 重ゲート:

1. **① 明示 TODO のみ抽出する**ことを backfill モード本文で明示（この節の宣言がそれ）。「② 次タスク候補」「③ ノート分析で出た課題」は**本モードで抽出しない**。
2. プロンプト末尾で `(c) backfill では ① 明示 TODO ラベル必須` と確認する（候補ごとに ① のラベル付与を強制）。
3. 抽出結果に対し、**`②`/`③` のラベル**を含む候補は提示前に破棄する（機械的封鎖。ラベル ②／③ が混入したら候補リストから外す）。

通常 (a)(b) モードでは 3 種ラベルは識別目的の必須（後段の grep / 承認ゲート判定に使う）。backfill のみ「① のみ通す」フィルタが追加で乗る形。**ラベルは作成後のタスクノートには残さない**（抽出時の制御用で完結）。

ユーザが過去ノートから ②③ を意図的に拾いたい場合は、backfill ではなく**対象ノートの抜粋を `input/` に書き出して input/ drain (b) モードで再抽出する**経路を取る（input/ 経由なら出所中立化されて (b) の通常規律で扱える）。日次蒸留 (a) の対象は「このセッションで触れたノート + カーソル以降の新規 pages」と定義済みで、過去ノートを「今日改めて読む」のは (a) の射程外なので含めない。backfill の 3 重ゲートは absolute・解除しない＝過去日の感覚を想像で埋める捏造リスクを取らない既定方針。

## やってはいけないこと

- 弱い繋がりから洞察を捏造する（A を B に水増しする）
- タイトルに解・抽象的な徳・実装ジャーゴン・造語を据える（要点を埋もれさせる。避けたい失敗を plain な確立語で名指す。背景: 「タイトルの付け方」節）
- 作業ログ・調査記録のタイトルを無理に抽象化・短縮する（具体性が消えると何を調べたか辿れなくなる＝実害）
- 気づきノードを作ってリンクを張らず orphan のまま置く（失敗は重複でなくリンク欠如。重複生成はむしろシグナル）
- 重複を恐れて気づきの捕捉・ノード化・リンクを省く（迷ったら分けて作りリンクする）
- 気づきを journal の1行 bullet に溜めて終わる（死文化する。ノード化して概念へリンクする）
- 事実・調査結果（検証で確定した客観的事実・仕様）に `#気づき` を付ける（気づきは踏んで得た主観的な学び・判断・方針。事実は外部検証可能な客観。背景: 冒頭 A／事実 区別）
- 気づきノードに `#気づき` を付け忘れる（事実・洞察と混ざり、調査記録と区別できなくなる）
- pages/ ノートを作成・改変したのに `updatedAt`（ISO-T）を打たない（recency 索引＝MOC の現状表示・並び替えが欠落する）
- 気づきノート（や当日触れた pages）に `## 更新履歴` の当日 `[[日付]]` リンクを付け忘れる（daily 被リンク集約に載らず日次記録から欠落する）。journal 日付リンクを `## 更新履歴` に集約せず本文に散らすのも不可（`FROM` が編集日とずれる）
- `input/` の原本を処理後も放置する（`archive/input/` へ退避して inbox を空に保つ。残量＝未処理キュー）
- drain 昇格で input 名が変わる／分割されるとき、元 input を指す既存リンクを `rg -l` で洗わず放置する（同名昇格を既定前提にしない＝リンク欠落。input/ drain モード節参照）
- 処理済み input を archive へ mv する前に `ls` 差分で件数照合せず、Sync 消失を見逃す（mv 前にスナップショット照合、欠落は止めて報告。input/ drain モード節参照）
- `imports/kindle/` `imports/wallabag/` の編集・リネーム
- 洞察ノートを専用フォルダに隔離する（置き場は `pages/` 直下。区別はタグ `#洞察` で行う＝過去の projects/ フォルダ失敗を繰り返さない）
- タスクノートを専用フォルダに隔離する（タスクも同じく `pages/` 直下。区別はタグ `#タスク` ＋ `progress:` 値で行う）
- タスクから直接洞察を立てる（層が違う＝弱い繋がりからの捏造になる。タスク → 気づき → 洞察 の経由必須）
- backfill モードで ② 次タスク候補・③ ノート分析で出た課題を抽出する（過去日の感覚を想像で埋める捏造を予防。① 明示 TODO のみ通す 3 重ゲート）
- タスクノートの `status:` を `done` に書き換える（タスクの完了は `progress: done` で表す。`status:` はノートライフサイクル＝触らない）
- 抽出時のラベル ①②③ を作成後のタスクノートに残す（frontmatter にも本文にも書かない）
- タスクノートの `## やること` を `- [ ]` チェックボックスで書く（plain な `- ` 箇条書きにする）
- タスクの `progress:` フィルタを daily の掲載クエリに混ぜる（誕生日記録の不変性を保つ）
- タスクノートに `#タスク` タグを付け忘れる（MOC/タスク.md と daily「今日生まれたタスク」の Dataview 集約から欠落する）
- 既存ノートへ無関係なリンクを大量挿入する（繋がりが実在するものだけ）
- `MOC/洞察.md` への定型的な手動追記（Dataview が集約する）
- 日次蒸留の完了時にカーソル `最終日次蒸留:` を更新し忘れる（次回が同じ範囲を二重に拾うか、対象選定が前回時刻のまま膨らみ続ける。ステップ5で実行時刻へ更新する）
- 運用ログを形骸化させる（重くして書かなくなる＝自動化フェーズの教師データが欠落する。最小テンプレ・承認ループ副産物に留め、件数が増えて重いなら粒度を落とす判断を運用ログ自身で観測する）
