# develop: 開発オーケストレータ

計画から実装、レビュー、修正、レポートまでの 5 フェーズを順次回す。各フェーズは可能な限り既存スキルに委譲し、無いものだけ Agent tool で subagent を起こす。

## フロー

### 1. 計画

- Skill tool で `plan` スキルを起動する。
- 戻りの plan.md パスと、main context に残った決定事項のニュアンスを後段で利用する。

### 2. 実装

- Skill tool で `implement` スキルを起動する。plan.md パス＋計画の主要な決定事項のサマリを渡す。
- `implement` が実装担当 subagent の起動・テスト実行・plan 突合の self-review までを担う。TDD 志向の切り替えも `implement` に委ねる。
- 戻りで `{変更概要, 変更ファイル一覧, 検証結果}` を受け取り main context に保持する。
- 注: `Skill` tool は main 会話で動くため、隔離されるのは `implement` が起こす実装 subagent のコード編集工程だけ（部分分離）。実装の全工程まで隔離したいときは、`Skill: implement` ではなく `Agent` で「implement を使う subagent」を起こす。

### 3. レビュー

- Skill tool で `code-review` スキルを起動する。
- effort level は依頼の重要度に合わせる（デフォルトは省略 = medium、厳重に見たいときは high/max、深掘りしたいときは ultra）。
- 指摘リストを受け取り main context に保持する。
- **指摘＋修正を 1 ステップで済ませたいとき**は `code-review --fix` を呼び、ステップ 4 を統合してよい（**この場合ステップ 4 の修正適用はスキップ**だが、ステップ 4 の再レビュー収束は `--fix` 経路でも行う。修正の透明性を上げたい場合は分割のまま）。
- ステップ 2 の `implement` 内 self-review とは観点で住み分ける（self-review＝plan 突合＋テスト green、ここ＝correctness/quality の adversarial レビュー。典拠は各 description）。同じバグ探しを二重にしない。**self-review をスキップさせたら step3 は必須**（両方抜けると検証ゼロ）。
- **成果物が非コード（プロンプト/設定等）なら、diff 前提で素通りせず対象を明示パスで読ませる**（diff が無いと code-review が空振りする）。

### 4. 修正と再レビュー収束

- レビュー指摘があれば Agent tool で修正担当 subagent を起動:
  - `subagent_type`: `general-purpose`
  - `prompt`: 指摘リスト＋plan.md パスを渡し、「指摘に対応してください。**修正後、修正したファイルを読み直し、(a) 各指摘に対応できているか (b) 他を壊していないか（regression）を自己確認し、対応概要に加えて自己確認結果も返してください**」（修正の自己申告だけを信じず、自己検証を必須にする。客観確認は下のループで担保する）。
- 軽微なクリーンアップが中心なら `simplify` スキルでも代替可能（simplify＝品質のみ・バグは見ない）。
- 指摘が無ければスキップ。前ステップで `--fix` を済ませた場合は修正適用はスキップするが、`--fix` で修正済みでも解消確認のため**最低 1 回は再レビューし、指摘が残れば通常経路と同じく最大 3 ラウンドまで**回す。
- ステップ 2 の `implement` 内修正とはトリガーで住み分ける（implement 内＝自分のテストを緑にする/plan 逸脱を直す、ここ＝code-review 指摘への対応）。
- **再レビュー収束ループ（検証の閉路）** — 修正後はステップ 3 を再実行し、修正が指摘を解消したか・新たな問題（regression）を入れていないかを確認する。「再レビュー→残指摘の修正」を**指摘ゼロになるか最大 3 ラウンドのいずれか早い方まで**回す。各ラウンドで客観確認（テスト green の再確認）も省かない。**3 ラウンド回しても指摘が残る場合は黙って打ち切らず、残る指摘をユーザに報告**してから次へ進む（自動収束の限界を隠さない。修正は書き込み工程なので、未検証のままステップ 5 へ流さない）。

### 5. レポート

- Agent tool でレポート担当 subagent を起動:
  - `subagent_type`: `general-purpose`
  - `prompt`: plan.md・実装サマリ・レビュー結果・修正サマリから作業レポートを **Obsidian の `~/workspace/notes/obsidian/Life/input/`** に書く（呼び出し時に明示パスがあれば優先）。内部で `obsidian:obsidian-cli` 等の既存スキルを活用してよい
- レポートの出力パスをユーザに報告する。

## ガード

- `/plan` の承認が取れる前に実装系 Agent やレビュー skill を起動しない。
- **副作用（chezmoi apply・commit・push・外部公開を伴う書き込み）の前は、plan 承認とは別に承認を取る**（plan 前ガードだけでは後半フェーズの不可逆操作を止められない。Obsidian 等の私的ノート出力は外部公開でないので対象外）。
- 各フェーズの結果（plan.md パス・変更概要・指摘リスト・修正概要）を main context に残し、次フェーズの prompt に組み込んで引き継ぎを担保する。
- **フェーズ間のユーザ確認は既定 ON。単一ファイル等の明確な小タスクのみ通しで回す。**通しでも動作検証（テスト green の客観確認）は省かない。
- **メタタスク（改修対象が SKILL／プロンプト文書）でも、ステップ 4 の再レビュー収束ループは本来の規約（最大 3 ラウンド）どおり回す。** 対象の失敗パターンが develop 側で再生産される懸念があっても予防的に打ち止めない（予防的な打ち止めは上流 skill 改善の不信任と等価）。実発生したらユーザと合意の上で下流 skill (plan / code-review) の改修タスクを起票する。**失敗接地**: 2026-06-10、plan SKILL で観点インフレ observed。**検証実績**: 2026-06-11、蒸留スキル（現 /harvest）改修で本来のループが正常収束（Round 2 で指摘ゼロ）。

## オプション

- ステップ 5 の前に `verify` スキル（**コード変更の動作確認用。Obsidian ノート検証の同名 `verify` とは別物**）で動作確認を挟むと、レポートに「動作確認済み」と書ける（依頼の性質に応じて選択）。
