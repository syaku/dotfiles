# 命名訂正事例集（drain 命名ゲート few-shot）

drain の気づき・洞察命名で人ゲートが訂正した推奨・訂正ペアの代表例（`notes/distill運用ログ.md` から人手蒸留した静的スナップショット）。抽出 agent（kizuki-extract / insight-detect）が spawn 時に Read し、初回命名・SendMessage 再命名・軸再考の各段で参照する。

## 使い方の指針（2 点）

1. **倣うのは訂正の方向（何が指摘され、どう直ったか）であって主張内容ではない。** 事例の主張をなぞって似たタイトルを作らない——内容は目の前の素材から立てる（insightPrompt【手本の使い方】と同型・過適合防止）。
2. **命名の確定前に「別の cycle で観察したらどう書くか」を自問する。** 今回の作業 cycle の固有語彙（ツール名・機構名・その cycle で取った対策の語彙）は、cycle 内に居ると一般語に見える。同じ機序を別の cycle で観察したときにも通じる語で書けているかを確かめてから提出する。

## 事例（種別・初稿 → 最終・訂正理由）

### 軸は一般化済みなのにタイトルで退化した型

- **気づき**: 初稿 `出力に残らない中間過程は人ゲートで審査されない` → 最終 `出力に残らない中間過程は審査されない`
  訂正理由: lesson_axis は審査主体を「外部レビュアー」と一般化済みなのに、タイトルだけ「人ゲート」に退化していた。審査主体を問わず成立する軸をタイトルに反映する。
- **気づき**: 初稿 `workflow script 経由の subagent は CLAUDE.md @import を継承しない` → 最終 `親プロセスの常時ロード規範は別プロセス起動の subagent に届かない`
  訂正理由: generalization_check に「別プロセス経由の subagent spawn 一般に置換可能」と書いたのにタイトル未反映のまま提出した。実装固有語（workflow script・CLAUDE.md・@import）を抜き、主語を親プロセス側に上げる。
- **気づき**: 初稿 `抽象化の効果は入力フィールドの参照先で止まる` → 最終 `次の処理が古いままだと前の修正は反映されない`
  訂正理由: 「抽象化」は具体例の 1 つで核ではない。lesson_axis の字面に引きずられず、本質（下流が古いままだと前の修正が反映されない）を平易な日常語で言い切る。

### cycle 語彙バイアスの多段抽象化（洞察）

- **洞察**: 初稿 `症状の連鎖は責務集約で畳む` →（「役割」軸への言い換えも特殊解と再指摘）→ 最終 `直しが続くなら前提を見直す`
  訂正理由: 「責務」「役割」「集約」は当該 cycle で取った対策の語彙で、cycle 内では一般語に見える。事例独立の軸（前提の見直し）まで 2 段の抽象化を重ねて到達した。
- **洞察**: 初稿 `subagent 規律の届く範囲は配置でなく呼び出し経路で決まる` →（`規範の届く範囲は配置と実行経路で分かれる` 等の中間案を経て）→ 最終 `親プロセスの情報は子プロセスに継承されるとは限らない`
  訂正理由: subagent・規律（rules の言い換え）・起動経路（実装依存語）が残る限り抽象不足。中間抽象の語彙を並べず、確立概念（親プロセス／子プロセス／継承）に閉じた形が最も転用が効く。

### 事実形から観点形への転換（洞察）

- **洞察**: 初稿 `段間の情報伝達は両端フィールドで決まる` → 最終 `中間処理を入れたら出口と入口の対応を見る`
  訂正理由: 「〜で決まる」の事実観察形では「次にどう振る舞うか」が見えない。レビュー・設計に使える観点形へ転換する（観察形でも判断軸として使えれば成立するが、この例は観点形が要点を立てた）。

### 堅い書き言葉の話し言葉化（気づき）

- **気づき**: 初稿 `短期間で書き直す予定のコードの重複は抽出を急がない` → 最終 `すぐ書き直すコードのダブりは無理にまとめない`
  訂正理由: 堅い書き言葉（重複・抽出）を話し言葉に。話し言葉化すると近接ノートとの機序一致も見えやすくなる。
- **気づき**: 初稿 `1 cycle に詰めた重い構造変更は fix 自体が defect を生む` → 最終 `重い変更を一度に詰め込むと直しが別のバグを呼ぶ`
  訂正理由: 英語混入（fix・defect）と堅い言い回しを、日常語の言い切りに直す。
- **気づき**: 初稿 `実害観測のない予防修正は scope を膨らませる` → 最終 `困ってないうちに先回りで直すと話が大きくなる`
  訂正理由: jargon（scope）と漢語の連なりを話し言葉に。何が起きるかが直接読める形にする。

### 英語混入の最小置換

- **気づき**: 初稿 `コードコメントの進行中タスク参照は完了後に rot する` → 最終 `コードコメントの進行中タスク参照は完了後に腐る`
  訂正理由: 英語混入は要点を保ったまま最小置換（rot → 腐る）で直す。タイトル全体を作り直さない。

## keep-in-sync 方針

- このファイルが命名訂正事例の**唯一の正本**。drain SKILL.md（4.1〜4.4・step 5）と workflow `harvest-pipeline.js` の `insightPrompt` からは**パス参照のみ**（本文を転記しない）。
- 更新は静的キュレーション——`notes/distill運用ログ.md` の追記に自動追従せず、drain 改修 cycle で観測に基づき人手で見直す。
- 訂正理由の方向が workflow の NAMING 定数群（`NAMING_FOR_KIZUKI` / `NAMING_FOR_INSIGHT`）・checker 基準と食い違ったときは NAMING 側を正とし、事例側を見直す。
