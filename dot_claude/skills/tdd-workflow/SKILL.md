# TDD ワークフロー（言語非依存）

このスキルは、テストを先に書いてから実装を進めるための作業指針を与える。
特定の言語・テストフレームワークに依存しない。コード例は擬似コードで示すので、
**プロジェクトで実際に使われているテストランナーの記法に置き換える**こと。
まず対象リポジトリのテスト構成（既存のテストの置き場所・命名・実行コマンド）を確認し、それに倣う。

## いつ使うか

- 新しい機能や関数を追加するとき
- バグを修正するとき（まず失敗する再現テストを書く）
- リファクタリングするとき（既存のテストを安全網にする）

## 基本サイクル: Red → Green → Refactor

1. **Red** — 期待する振る舞いを表すテストを1つ書く。実装はまだ無いので失敗する。失敗することを実際に確認する（テスト自体のバグを早期に弾く）。
2. **Green** — テストを通す最小限の実装を書く。きれいさより「通すこと」を優先。
3. **Refactor** — テストが緑のまま、重複の除去・命名改善・整理を行う。各ステップ後にテストを再実行。

小さく回す。1サイクルで扱う振る舞いは1つに絞る。

## バグ修正の手順

1. バグを**再現する失敗テスト**を先に書く。
2. そのテストが通るように修正する。
3. テストはそのまま残す（回帰防止）。

「直った」だけで終えず、再現テストをコミットに含める。

## テストの書き方

### 振る舞いを検証する（実装詳細をテストしない）

内部状態・private な構造・呼び出し順序ではなく、**入力に対する出力／外部から観測できる効果**を検証する。実装を変えてもテストが壊れないなら、それは良いテスト。

```
# 悪い: 内部カウンタを直接覗く
assert obj._internal_count == 5

# 良い: 公開された振る舞いを確認する
assert render(obj) contains "Count: 5"
```

### Arrange-Act-Assert（準備・実行・検証）

各テストを3段に分けて読めるようにする。

```
test "空クエリは空リストを返す":
    parser = new Parser()          # Arrange
    result = parser.parse("")      # Act
    assert result == []            # Assert
```

### テストは独立させる

各テストが自前でデータを用意し、後始末する。テスト間で状態を共有しない／実行順序に依存しない。

### 何を確認するか

正常系（happy path）だけでなく、**境界とエラー経路**を必ず含める。

- 境界値: 空・null/None・0・1件・最大件数・上限超過
- エラー: 不正入力・依存先の失敗・タイムアウト
- 外部依存（ネットワーク・DB・時刻・乱数）はテスト境界で**モック/スタブ**して、ユニットテストを決定的かつ高速に保つ

### 命名

テスト名は「何を確認するか」を述べる。`test_1` ではなく `空クエリは空リストを返す` のように、失敗時にログだけで原因が分かる名前にする。

## アンチパターン

- **脆いセレクタ/前提**: 偶然の実装詳細（内部クラス名、CSSクラス、メッセージ文字列の完全一致）に依存しない。
- **テストのためのテスト**: カバレッジ数値を埋めるだけの無意味なアサートを書かない。
- **巨大テスト**: 1テストで多くを検証しすぎない。失敗時に原因が特定できる粒度に分ける。
- **happy path だけ**: エラー経路を書かないテストは安全網にならない。

## カバレッジの扱い

カバレッジは「テストが手薄な箇所を見つける**信号**」であって、達成すべき**目標値ではない**。
低い行は要注意だが、100% は品質を保証しない。数値合わせのテストを書くくらいなら、
重要な振る舞い・分岐・エラー経路が網羅されているかを目で確認する方を優先する。

## 進め方のチェックリスト

- [ ] 実装より先にテストを書いた
- [ ] そのテストが（実装前に）失敗することを確認した
- [ ] 最小実装で緑にした
- [ ] 緑を保ったままリファクタした
- [ ] 境界とエラー経路のテストがある
- [ ] テストは独立・決定的・高速
- [ ] スキップ/無効化したテストを残していない
