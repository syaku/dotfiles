# TDD ワークフロー（言語非依存）

テストを先に書いてから実装を進めるための作業指針。言語・テストフレームワーク不問。コード例は擬似コードで示すので、**プロジェクトで実際に使われているテストランナーの記法に置き換える**こと。

## 起動時前準備（必ず実行）

依頼に着手する前に、対象リポジトリで以下を確認する:

1. **テスト構成の把握** — Glob で `**/*test*` `**/*spec*` `**/tests/**` 等を探し、既存テストの置き場所・命名規約・実行コマンドを掴む。`package.json` / `pyproject.toml` / `Cargo.toml` / `Makefile` / `justfile` 等を Read してテストランナーと実行コマンドを特定する。
2. **既存テストが green か確認** — 既存のテスト一式を実行して全部 pass することを確認する。red が混じっているなら依頼者に共有してから進めるかを判断する。基準点が壊れているとサイクルが回らない。
3. **TDD が適切か判断** — 後述「使わない場面」に該当しないか一瞬考える。該当するなら依頼者に切り替えを提案。

ここを飛ばすと「想定外のテスト構成」「既存壊れていた」で後段が崩れる。

## いつ使うか / 使わないか

### 使う

- 新しい機能・関数を追加するとき
- バグを修正するとき（まず失敗する再現テストを書く）
- 安全網が欲しいリファクタリング（既存テストが薄ければ characterization テストを先に書く）

### 使わない（無理に当てはめると害になる）

- 探索的なスパイク・PoC（捨てるコード）
- UI のレイアウト微調整・コピー変更
- ライブラリのバージョン上げ・config 変更
- 純粋な glue/wiring（テストするほどのロジックがない）
- マイグレーションや一回限りのスクリプト

判断に迷ったら「このコードの**振る舞い**を文章で説明できるか？」を問う。説明できるならテストにできる。

## 全体フロー

### 0. テストリストの作成

着手前に、依頼を**検証可能な振る舞いの単位**に分解し、テストリストを書き出す（コメント・TODO・箇条書きでよい）。粒度の目安: 1 テストで 1 振る舞い。

例（パーサのバグ修正）:

```
- [ ] バグを再現する失敗テスト（修正前は落ちる）
- [ ] 境界: 空入力で例外を出さない
- [ ] エラー: 不正フォーマット時に明示メッセージで失敗
- [ ] 回帰: 既存の正常系が壊れていないこと
```

リストは作業中に追加・削除してよい。「次に何を書くか」を常にリストから選ぶ。

### 1. Red — 失敗テストを 1 つ書く

- リストの先頭から 1 件選び、**期待する振る舞いを表すテスト**を書く。
- 実装はまだ無いので失敗するはず。**実行して、想定どおりのメッセージで失敗することを確認**する。
  - 「実行せずに red のはず」で進めない。テスト自体のバグや setup の誤りを早期に潰す。
  - コンパイルエラー・import エラーで落ちているのは red ではない。テストランナーが「アサート失敗」で落ちる状態まで持っていく。

### 2. Green — 通す最小実装

以下の 3 戦略から状況に応じて選ぶ:

- **Obvious Implementation** — 実装が自明なときはそのまま書く。
- **Fake It** — まずハードコード値で通す。次のテスト追加で一般化を強いる。設計に迷うときに使う。
- **Triangulation** — 別ケースのテストを追加し、両方を通す一般化を導く。型・分岐の境界が曖昧なときに使う。

きれいさより**通すこと**を優先。設計の改善は Refactor フェーズで行う。

### 3. Refactor — 緑のまま整理

- 重複の除去、命名改善、構造の整理。
- **「2 つの帽子」**: refactor 中は新規振る舞いを追加しない。逆も同じ（green 取得中に refactor しない）。
- 各小さな変更ごとにテストを再実行。red になったら直前の状態に戻す。
- 整理が終わったらコミット。

### 4. リストの次へ

完了した項目をチェックし、リストの次に進む。リストが空になったら依頼完了。途中で新しい振る舞いに気づいたらリストに追記する。

## テストの書き方

### 振る舞いを検証する（実装詳細をテストしない）

内部状態・private 構造・呼び出し順序ではなく、**入力に対する出力／外部から観測できる効果**を検証する。実装を変えてもテストが壊れないなら良いテスト。

```
# 悪い: 内部カウンタを直接覗く
assert obj._internal_count == 5

# 良い: 公開された振る舞いを確認する
assert render(obj) contains "Count: 5"
```

### Arrange-Act-Assert

```
test "空クエリは空リストを返す":
    parser = new Parser()       # Arrange
    result = parser.parse("")   # Act
    assert result == []         # Assert
```

各テストを 3 段に分けて読めるようにする。Assert は原則 1 テスト 1 つ（複数になるなら分割を検討）。

### 独立・決定的・高速

- 各テストが自前でデータを用意し、後始末する。実行順序に依存しない。
- 時刻・乱数・ネットワーク・DB など外部依存はテスト境界で**スタブ/モック/フェイク**する。
- 実行が遅いテストは TDD のサイクル速度を壊す。重い統合テストは別レイヤに分け、TDD では薄いユニットを高速に回す。

### 命名

`test_1` ではなく `空クエリは空リストを返す` のように、**失敗ログだけで原因が分かる名前**にする。プロジェクトの命名規約があればそれに従う。

### 何を確認するか

正常系（happy path）だけでなく、**境界とエラー経路**を必ず含める:

- 境界: 空・null/None・0・1 件・最大件数・上限超過・型境界（負数、非 ASCII 等）
- エラー: 不正入力・依存先の失敗・タイムアウト・並行アクセス

## バグ修正の手順

1. 不具合を**再現する失敗テスト**を書く（境界・前提条件を含めて）。
2. 失敗を確認してから修正する。
3. テストはそのまま残す（回帰防止）。
4. 同じ根本原因で他に波及しないか確認し、必要なら追加テスト。

「直った」だけで終わらせない。再現テストをコミットに含める。

## レガシーコードへの適用

既存テストが無い／薄いコードを触るときは:

1. **Characterization テスト** — 今の挙動（バグも含めて）を固定するテストを書く。理想ではなく**現状**を写す。
2. これを安全網に refactor。
3. refactor 後、正しい振る舞いを表すテストを追加し、実装を修正。

Seam（接合点）を見つけて、最小の改変でテスト可能にする。最初から完全なテストを揃えようとしない。

## モックの使い分け

- **Stub** — 戻り値を固定して呼び出し側のロジックを検証
- **Mock** — 呼び出された事実・引数を検証（最小限に。やりすぎると実装結合）
- **Fake** — 軽量な実装で代替（インメモリ DB 等）

**境界でモックする**。アプリ内部の小さなコラボレータをモックしすぎると、実装に結合したテストになり refactor 耐性が落ちる。原則は「シナリオの境界」（HTTP・DB・時刻・ファイルシステム・乱数）。

## アンチパターン

- **脆いセレクタ/前提**: 偶然の実装詳細（内部クラス名、CSS クラス、メッセージ完全一致）に依存しない
- **テストのためのテスト**: カバレッジ数値を埋めるだけの無意味なアサート
- **巨大テスト**: 1 テストで多くを検証しすぎる。失敗時に原因特定できる粒度に
- **happy path だけ**: エラー経路を書かない安全網は安全網ではない
- **モック過多**: 内部呼び出しを全部モックすると実装変更で全壊する
- **テスト skip 放置**: `xit` / `skip` / `xfail` のまま戻さない
- **green の仮装**: 進めたさで `assert True` 等で通すなど

## 詰まったとき

- **次のテストが書けない** → 設計粒度が大きすぎる。1 段小さい振る舞いに分解し直す
- **red にできない** → コンパイル/import エラーで落ちていないか確認。アサート段階で落ちる状態まで持っていく
- **テストが過度に複雑** → 実装側の依存が多すぎる兆候。インターフェース分離やコラボレータ削減を検討
- **3 回連続で red 解消に失敗** → 推測を止めて debugger / print / 既存テストの実装を読む。前提にしている像が誤っているシグナル

## コミット粒度

- Green 到達時 / Refactor 完了時にコミット。
- red 状態のコミットは原則作らない（CI を red にしない）。WIP は別ブランチや stash で逃がす。
- 1 コミット = 1 振る舞いの追加か、1 つのリファクタ。混在させない。

## カバレッジの扱い

「テストが手薄な箇所を見つける**信号**」であって、達成すべき**目標値ではない**。低い行は要注意だが 100% は品質を保証しない。数値合わせのテストを書くくらいなら、重要な振る舞い・分岐・エラー経路が網羅されているかを目で確認する方を優先する。

## 完了チェックリスト

- [ ] 起動時前準備（テスト構成把握・既存 green 確認）が済んでいる
- [ ] テストリストを作って進めた
- [ ] 各テストが（実装前に）正しいメッセージで失敗することを確認した
- [ ] 最小実装で緑にした
- [ ] 緑を保ったままリファクタした
- [ ] 境界とエラー経路のテストがある
- [ ] テストは独立・決定的・高速
- [ ] skip/無効化したテストを残していない
- [ ] レガシー領域なら characterization テストで現状を固定してから触った
