# skill-review: develop オーケストレータの評価

develop スキル（計画→実装→レビュー→修正→レポートの5フェーズを既存スキルへ委譲するオーケストレータ）の改善点を、繰り返し当てられる評価の型として洗い出すスキル。**当面は develop 専用に固定**し、汎用の引数化はしない（評価対象が2つ目に増えた時点の YAGNI 解除タスクに切り出す。末尾「展開時の差し替え点」参照）。

評価対象が小さい（develop 39行＋委譲先数枚＋トレース1セッション）ため、**読み取り系を1 read-only subagent に統合**し、develop と同型の多段オーケストレータは建てない（批評対象の複雑性を仕組み側に持ち込まない）。subagent は実質1系統（フェーズ1）に圧縮し、実走（フェーズ2）はユーザ承認の上で別建てにする。

評価は3ソース × 3軸で行う。**3軸は全フェーズを通して固定**で、これ以外に広げない:

- **フェーズ設計・委譲構造** — フェーズ分割の妥当性、委譲先スキルとの住み分け、引き継ぎ（context の受け渡し）。
- **プロンプト・指示の品質** — 各フェーズ／subagent への指示の曖昧さ・冗長・抜け。
- **失敗・抜け穴の堅牢性** — ガード破り、データ／context ロスト、暴走の余地。

## フロー

### 1. 静的レビュー＋トレース分析（単一 read-only subagent）

Agent tool で read-only の subagent を1つ起こす（`subagent_type`: `Explore` または `general-purpose`）。評価対象・委譲先・トレースを**変更せず読むだけ**で、3軸の批評を根拠付きで返させる。prompt には以下 (a)(b)(c) を全て含める。

- **(a) 静的レビュー** — develop の SKILL.md と、**精読可能な委譲先**を読む:
  - `~/.claude/skills/develop/SKILL.md`（評価対象本体）
  - `~/.claude/skills/plan/SKILL.md`・`~/.claude/skills/implement/SKILL.md`（委譲先・精読可）
  - プラグイン配下の `verify`（`~/.claude/plugins/marketplaces/agenticnotetaking/skill-sources/verify/SKILL.md`、委譲先・精読可）
  - **`code-review`・`simplify` は SKILL.md が無い**（ビルトイン）。精読できないので、develop 側の呼び出し記述＋スキル description のみで評価し、**「SKILL.md 不在で精読不可」をレポートに明記**させる（委譲構造軸の評価範囲の穴を隠さない）。
- **(b) トレース分析** — develop の実走セッションを読み、詰まり・無駄・フェーズ間ロスト・ガード破りを抽出する:
  - **対象セッションは固定せず動的に解決する**: `~/.claude/projects/-Users-syaku-workspace/` 配下の `*.jsonl` を `Launching skill: develop` で Grep し、ヒットした全セッション（メイン `<session>.jsonl` ＋ `<session>/subagents/agent-*.jsonl`、jsonl は1行1イベント）を対象にする。既知の初回トレースは `30d0b52b-*` だが、それ以降の develop 実走（この評価を回したセッション自身を含む）も自動で拾う。session ID をハードコードすると古いトレース1件しか読まず、「繰り返し当てられる型」という目的を自分で損なう。
  - **トレースが少数のうちは、実運用の統計ではなく「ケーススタディ」と正直に位置づける**よう指示に明記する（件数が少なければ「N 件のケーススタディ」と書き、統計的一般化をしない）。
- **(c) 3軸で批評** — (a)(b) を上記3軸（フェーズ設計・委譲構造 / プロンプト・指示の品質 / 失敗・抜け穴の堅牢性）で批評させる。**各指摘に根拠（`SKILL.md:行` か trace のイベント）の併記を必須**にする（grounding。空中戦・hallucination 防止。根拠の無い指摘は返さないよう prompt で明示する）。深刻度案は、下記フェーズ3の3段ルーブリック（高＝ガード破り・データ／context ロスト／中＝指示の曖昧さ・冗長・住み分け不明／低＝文体・可読性）を subagent prompt に渡して付けさせる。

戻りは `{軸, 指摘, 根拠, ソース（静的／トレース）, 深刻度案, 改善提案}` の素材リストとして受け取り、main context に保持する。

### 2. 実走観察（別建て・初回は推奨）

性質（実装系スキルの起動・ユーザ承認・コスト）がフェーズ1と違うため別フェーズにする。**skill-review 本体からは自動実行しない。** ユーザ承認の上で、小規模なサンプルタスクを develop で1回実走させ、新規トレースを作ってフェーズ1(b) に還元する（2件目のケーススタディを得る経路）。

- トレースが薄いうちは、**初回適用ではこのフェーズを「既定で提案」する**（任意に倒しきらず、やる前提でユーザに諮る）。
- 将来トレースが貯まれば省略可。
- 実走で得た新規トレースは、フェーズ1(b) の入力に追加して再度静的＋トレース批評に掛ける。

### 3. 統合（畳み込みルール）

フェーズ1（＋実施したならフェーズ2）の指摘を1つの改善点レポートに畳む。

- **統合前のスポットチェック（grounding 検証）** — 畳む前に、指摘から2〜3件を選び、併記された根拠（`SKILL.md:行` か trace 事象）を**実際に開いて指摘を支持するか確認**する。支持されない（hallucinated line・牽強付会な引用）指摘は落とすか「要確認」マークを残す。grounding は finder への要求だけでは担保されず、再照合が要る。
- 指摘を `{軸, 指摘, 根拠, ソース, 深刻度, 改善提案}` の表にする。
- **重複の畳み込み** — 「同一軸かつ同一の根拠箇所／同一の挙動を指す」ものを重複とみなし、1行に畳む。
- **ソース食い違いの扱い** — 静的では潜在的懸念だがトレースでは未顕在、のような食い違いは**消さず** `ソース` 列に併記する。**トレースで顕在化した（実際に起きた）ものは深刻度を1段格上げ**する。
- **深刻度ルーブリック（3段）**:
  - **高** — ガード破り・データ／context ロスト。
  - **中** — 指示の曖昧さ・冗長・委譲の住み分けが不明。
  - **低** — 文体・可読性。
- **レポート冒頭にソースの重みを明示** — 既定の1回は**静的（主体）＋トレース（補助）の2ソース**、実走観察（フェーズ2）は承認時に加わる3つ目、と書く（description の「3ソース」を名目でなく実態に合わせる）。トレースが少数なら「N 件のケーススタディ」と件数も添える。委譲先のうち `code-review`・`simplify` が精読不可だった旨も評価範囲の穴として明記する。
- **出力先は呼び出し時に指定**できるが、**既定は Obsidian の `~/workspace/notes/obsidian/Life/input/`**（人＋AI 成果物の capture inbox。会話文脈に依らない生成物はまず input/ に着地し、後で `/distill` が `pages/` ノードへ昇格させる運用）。frontmatter は付けてよいが pages テンプレ強制はなく、本文 H1 は置かない。永続化を Obsidian にするのは `~/workspace/tasks/` が削除されうるため（`tasks/CLAUDE.md` 規範に整合）。中間作業物のみ `~/workspace/tasks/<slug>/` 配下に置く。呼び出し時に明示パスがあればそれで上書きする。

## ガード

- 評価対象のスキル本体・委譲先を**変更しない**。read-only の評価に徹する（改善の適用は別タスク）。
- フェーズ2（実走）を**ユーザ承認なしに自動実行しない**（実装系スキルの起動・コスト・暴走防止）。
- 各指摘は**根拠（`SKILL.md:行` か trace イベント）併記が無ければ不採用**（grounding 必須）。
- 評価軸は3つ（フェーズ設計・委譲構造 / プロンプト・指示の品質 / 失敗・抜け穴の堅牢性）に固定する。トークン／コスト効率、委譲先スキル単体の改善、常設インフラ化は対象外。
- トレースが少ないうちは、フェーズ1(b) を統計的事実のように書かせない（1件のケーススタディとして位置づける）。

## やってはいけないこと

- 評価対象や委譲先のファイルを編集する（read-only 評価のはずが実装に踏み込む）。
- 根拠なしの指摘を載せる（hallucinated line・牽強付会な引用。指摘ごとに根拠箇所を実際に開いて支持されるか確認する）。
- develop と同型の多段オーケストレータに肥大させる（評価対象の複雑性を仕組み側に持ち込まない。読み取り系は1 subagent に統合する）。
- フェーズ2（実走）を承認なしに自動で回す。
- 精読不可（`code-review`・`simplify`）を黙って評価対象から落とす、または精読したかのように書く（評価範囲の穴はレポートに明記する）。
- 改善点レポートを `~/workspace/tasks/` 配下だけに残す（tasks は削除されうる。永続化は Obsidian）。
