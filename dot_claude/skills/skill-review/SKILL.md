# skill-review: 任意自作 skill の評価

自作 skill（計画→実装→…の多段オーケストレータでも、単発の処理 skill でも）の改善点を、繰り返し当てられる評価の型として洗い出すスキル。**評価対象は引数で受け取る**（後方互換は持たない）。対象 SKILL.md の内容から委譲先・subagent 起動・外部 IO を自動抽出し、必要メタ情報を skill 側に持たせない。

評価対象は概して小さい（SKILL.md 数十行＋委譲先数枚程度）ため、**読み取り系を1 read-only subagent に統合**し、評価対象 skill と同型の多段オーケストレータは建てない（批評対象の複雑性を仕組み側に持ち込まない）。subagent は実質1系統（フェーズ1）に圧縮し、実走（フェーズ2）はユーザ承認の上で別建てにする。

評価は3ソース × 3軸で行う。**3軸は全フェーズを通して固定**で、これ以外に広げない:

- **フェーズ設計・委譲構造** — フェーズ分割の妥当性、skill 委譲・subagent 起動・外部 IO（Bash curl／WebFetch／filesystem 直書きなど）の住み分け、引き継ぎ（context の受け渡し）。
- **プロンプト・指示の品質** — 各フェーズ／subagent への指示の曖昧さ・冗長・抜け。
- **失敗・抜け穴の堅牢性** — ガード破り、データ／context ロスト、暴走の余地。

## フロー

### 0. 対象 skill の確定

引数 `<skill-name>`（kebab-case 1 語）を必須で受け取る。

- **引数なしで起動された場合**: 「`/skill-review <skill-name>` を指定してください（例: `/skill-review develop`）」と告知して停止する。
- **形式バリデーション**: 受け取った引数は正規表現 `^[a-z][a-z0-9-]*$` に従う必要がある（先頭は小文字英字、以降は小文字英数字とハイフンのみ。先頭末尾ハイフン・連続ハイフン・空文字・大文字・全角・コロンを含む語は全て形式違反として弾く）。形式違反のときは「引数 `<received>` は kebab-case 1 語の形式に従いません（正規表現 `^[a-z][a-z0-9-]*$`）。**plugin namespace 形式（`plugin:skill` のようにコロンを含む語）は受け付けません — plugin skill は SKILL.md 精読不可で評価不能です**。typo の場合は再指定してください」と告知して停止する。形式 OK で初めて次の実在確認に進む。
- **対象 SKILL.md パスの解決**: `~/.claude/skills/<skill-name>/SKILL.md` を Read で実在確認する。
- **不在の場合のエラー（理由併記）**: 「対象 skill `~/.claude/skills/<name>/SKILL.md` が見つかりません。**対象は自作 skill に限ります（ビルトイン skill・plugin skill は SKILL.md 精読不可で評価不能。typo の場合は kebab-case 1 語で再指定してください）**」と告知して停止する。ビルトイン／plugin と typo が一律不在エラーで潰れないよう、メッセージで理由を分離する。
- **chezmoi 管理下の source/target 整合の確認**: 対象 SKILL.md は target（`~/.claude/skills/<name>/SKILL.md`）で解決するが、**chezmoi 管理下では target が source の apply 待ちで stale になりうる**。対象 skill が chezmoi 管理下で、最新の改修が target に反映されていない疑いがある場合は、`chezmoi diff <name>` 等で source と target の差分を確認してから評価せよ（差分があれば改修前の target を評価してしまい、改修の効能検証として無意味になる）。失敗接地: 2026-06-10、skill-review 自身の汎用化作業中、source 改修→target 未 apply の状態で `/skill-review skill-review` を回すと改修前 target を評価する潜在事故が code-review で発覚した。

解決済みの skill 名／SKILL.md パスを以降のフェーズで「対象識別子」として使う。本文中の `<対象 skill 名>` は解決済みの skill 名で読み替える。

**subagent prompt への引き継ぎ**: 解決済みの skill 名は、以降のフェーズで起動する subagent の prompt 内に**文字列補間で埋め込む**（本文中の `<対象 skill 名>` 表記は subagent prompt 組み立て時に実際の skill 名に置換する）。subagent 自身に skill 名を再解決させない（main 側で確定したものを引き渡し、解決の二重実装を避ける）。

### 1. 静的レビュー＋トレース分析（単一 read-only subagent）

Agent tool で read-only の subagent を1つ起こす（`subagent_type`: `Explore` または `general-purpose`）。評価対象・委譲先・トレースを**変更せず読むだけ**で、3軸の批評を根拠付きで返させる。prompt には以下 (a)(b)(c) を全て含める。

- **(a) 静的レビュー** — `<対象 skill 名>` の SKILL.md と、**精読可能な委譲先**を自動抽出して読む。subagent prompt に以下を明示する:
  - 対象 SKILL.md（`~/.claude/skills/<対象 skill 名>/SKILL.md`）を読む。
  - 対象 SKILL.md 本文から以下 3 種を抽出する: **(i) skill 委譲**（`Skill tool で X` 等の参照）、**(ii) subagent 起動**（`Agent` / `subagent_type` 言及）、**(iii) 外部 IO**（**`Bash`／`WebFetch` 等の外部呼び出し、`Edit`／`Write` による vault・workspace 等の永続化先への書き込み、`Read`／`Glob`／`Grep` による評価対象外ファイルの読み取り**を含む。WebFetch・Bash curl・filesystem 直書きなど）。
  - **抽出根拠の明示**: 抽出した各項目（skill 委譲先名／subagent 起動の有無／外部 IO の種類）には、**対象 SKILL.md の該当行番号＋逐語引用を必ず併記**せよ。引用文が『実際の呼び出し』を表すか『例示／注記／類似処理の対比』を表すかを 1 行で判別せよ。例示・注記・対比は委譲先に数えない（失敗接地: develop SKILL.md 行 22 注記の `Skill: implement ではなく Agent で…` のような文は『例示／対比』であり実際の委譲ではない）。
  - 各 skill 委譲先について、以下 2 系統で実在を判定する（どちらかに在れば精読可、両方とも不在＝精読不可（ビルトイン））:
    - **(i) 自作 skill**: `~/.claude/skills/<delegate-name>/SKILL.md` を Read または Glob で実在確認する。
    - **(ii) plugin 配下 skill**: `~/.claude/plugins/marketplaces/*/skill-sources/<delegate-name>/SKILL.md` を Glob で動的解決する（marketplace 名は固定しない）。
    - 失敗接地: 2026-06-10、汎用化改修時に plugin 配下を探索範囲から落とした初版を作り、develop 評価で `verify`（plugin 配下に SKILL.md 実在）が誤って精読不可扱いになるリグレッションが code-review で発覚した。plugin 配下の SKILL.md も精読可として扱う。
  - 精読可の委譲先 SKILL.md は読み込んで評価に使う。精読不可は呼び出し記述＋ description のみで評価し、**「SKILL.md 不在で精読不可」をレポートに明記**させる（委譲構造軸の評価範囲の穴を隠さない）。
  - **検算ハンドル**: **自動抽出した委譲先リスト（skill 委譲先名＋精読可否＋抽出根拠の行番号と逐語引用、subagent 起動の有無、外部 IO の種類）を、評価本体に入る前に中間出力としてレポート冒頭に必ず吐く**。これで対象 SKILL.md の実態と抽出結果が一致しているかを目視チェックできる。**中間出力を受け取った main context（オーケストレータ）側は、評価本体に進む前に抽出リストと対象 SKILL.md 実態を目視照合する。照合で違和感（例示と実呼び出しの混同・委譲先の漏れ）があれば subagent に再抽出を依頼**してから評価本体に進む。
- **(b) トレース分析** — `<対象 skill 名>` の実走セッションを読み、詰まり・無駄・フェーズ間ロスト・ガード破りを抽出する:
  - **対象セッションは固定せず動的に解決する**: `~/.claude/projects/*/**/*.jsonl` を `Launching skill: <対象 skill 名>` で Grep し、cwd を限定しない（jsonl path は `~/.claude/projects/<encoded-cwd>/` で cwd 依存のため、`-Users-syaku-workspace` 配下に絞ると他 cwd セッションのトレース（例: vault cwd で動いた distill）を取りこぼす）。ヒットした全セッション（メイン `<session>.jsonl` ＋ `<session>/subagents/agent-*.jsonl`、jsonl は1行1イベント）を対象にする。session ID をハードコードすると古いトレース1件しか読まず、「繰り返し当てられる型」という目的を自分で損なう。この評価を回したセッション自身も含めて自動で拾う。
  - **トレースが少数のうちは、実運用の統計ではなく「ケーススタディ」と正直に位置づける**よう指示に明記する（件数が少なければ「N 件のケーススタディ」と書き、統計的一般化をしない）。ヒットゼロなら「トレース未取得」と明記する。
- **(c) 3軸で批評** — (a)(b) を上記3軸（フェーズ設計・委譲構造 / プロンプト・指示の品質 / 失敗・抜け穴の堅牢性）で批評させる。**各指摘に根拠の「逐語引用」併記を必須**にする（grounding。空中戦・hallucination 防止）。行番号だけの併記は不可——`SKILL.md:行` なら**その行の本文をそのまま引用**させ、trace 主張なら**該当 jsonl イベントの逐語テキスト（要約でなく原文）＋ファイルパス**を引用させる。理由: finder は実在の行番号を引きつつ中身を捏造しうる（失敗接地: 2026-06-09、develop 評価で finder が L30/L52 を「ガード無し」と断じたが実ファイルには当該ガードが実在。行番号だけの根拠では「引用したが内容が指摘を支持しない」を検出できない）。逐語引用なら後段の再照合が機械的に効く。深刻度案は、下記フェーズ3の3段ルーブリック（高＝ガード破り・データ／context ロスト／中＝指示の曖昧さ・冗長・住み分け不明／低＝文体・可読性）を subagent prompt に渡して付けさせる。
  - **委譲構造軸の縮退ルール**: 対象 skill が委譲ゼロ（skill 委譲・subagent 起動なし、外部 IO のみで完結する単発型）の場合は「委譲構造: `N/A: 委譲先なし（subagent 起動・外部 IO のみで完結）`」と書かせ、軸を抹消はしない。フェーズ設計＝内部ステップの分割妥当性はどの skill にも問えるため、軸全体を落とさない。
  - **対話のみ skill の縮退ルール**: subagent 起動・外部 IO もゼロで Read/Edit のみで完結する **対話のみ skill** の場合は「委譲構造: `N/A: 対話完結（外部 IO・subagent なし）`」と書かせる。フェーズ設計・プロンプト品質・失敗抜け穴の 3 軸（うち委譲構造は N/A）として評価する（残り 2 軸で内部ステップの分割妥当性・指示品質・抜け穴は問える）。

戻りは `{軸, 指摘, 根拠, ソース（静的／トレース）, 深刻度案, 改善提案}` の素材リストとして受け取り、main context に保持する。

### 2. 実走観察（別建て・初回は推奨）

性質（実装系スキルの起動・ユーザ承認・コスト）がフェーズ1と違うため別フェーズにする。**skill-review 本体からは自動実行しない。** ユーザ承認の上で、小規模なサンプルタスクを `<対象 skill 名>` で1回実走させ、新規トレースを作ってフェーズ1(b) に還元する（2件目のケーススタディを得る経路）。

- トレースが薄いうちは、**初回適用ではこのフェーズを「既定で提案」する**（任意に倒しきらず、やる前提でユーザに諮る）。
- 将来トレースが貯まれば省略可。
- 実走で得た新規トレースは、フェーズ1(b) の入力に追加して再度静的＋トレース批評に掛ける。
- **サンプルタスクは副作用が小さいものを選ぶ**（read-only / 軽量な処理 / vault 内テストノートへの書き込み程度）。対象 skill が破壊的副作用（git push / chezmoi apply / 外部 API 課金 / 大規模ファイル削除）を持つ場合は、**実走をスキップし** 既存トレースのみでフェーズ 1 に閉じる。実走の有無はユーザ承認時に明示する。

### 3. 統合（畳み込みルール）

フェーズ1（＋実施したならフェーズ2）の指摘を1つの改善点レポートに畳む。

- **統合前の grounding 再照合** — 畳む前に、併記された逐語引用を**実際に開いて、引用が原文と一致し、かつ指摘を支持するか**を確認する。サンプルでなく次は悉皆: **全『高』指摘＋全トレース由来指摘は必ず再照合する**（中／低はサンプル可）。理由: 捏造は結論を動かす高深刻度・トレース主張に出やすく、サンプルだと見逃す（失敗接地: 2026-06-09、finder が高深刻度3件で実ファイルと矛盾する主張＋未読トレースの捏造事象を返した。2〜3件サンプルでは取りこぼしうる構成だった）。確認は「行番号が実在するか」でなく「**引用文が原文どおりで、その内容が指摘を実際に支持するか**」まで見る。支持されない（hallucinated line・原文と食い違う引用・牽強付会）指摘は破棄する（「要確認」で温存しない。grounding が崩れた指摘を残すと下流の判断を汚す）。
- 指摘を `{軸, 指摘, 根拠, ソース, 深刻度, 改善提案}` の表にする。
- **重複の畳み込み** — 「同一軸かつ同一の根拠箇所／同一の挙動を指す」ものを重複とみなし、1行に畳む。
- **ソース食い違いの扱い** — 静的では潜在的懸念だがトレースでは未顕在、のような食い違いは**消さず** `ソース` 列に併記する。**トレースで顕在化した（実際に起きた）ものは深刻度を1段格上げ**する——ただし**上の再照合で逐語イベントを裏取りできたものに限る**。裏取りできないトレース主張は格上げしないどころか破棄する（捏造されやすいトレース事象を格上げ規則が増幅する逆効果を防ぐ。失敗接地: 2026-06-09、develop 評価で finder の捏造トレース事象が「顕在化したから高」と格上げされかけた）。
- **深刻度ルーブリック（3段）**:
  - **高** — ガード破り・データ／context ロスト。
  - **中** — 指示の曖昧さ・冗長・委譲の住み分けが不明。
  - **低** — 文体・可読性。
- **レポート冒頭にソースの重みを明示** — 既定の1回は**静的（主体）＋トレース（補助）の2ソース**、実走観察（フェーズ2）は承認時に加わる3つ目、と書く（description の「3ソース」を名目でなく実態に合わせる）。トレースが少数なら「N 件のケーススタディ」と件数も添える。**自動抽出された精読不可委譲先（あれば）**は評価範囲の穴として明記する。
- **出力先は呼び出し時に指定**できるが、**既定は Obsidian の `~/workspace/notes/obsidian/Life/input/`**（人＋AI 成果物の capture inbox。会話文脈に依らない生成物はまず input/ に着地し、後で `/distill` が `pages/` ノードへ昇格させる運用）。frontmatter は付けてよいが pages テンプレ強制はなく、本文 H1 は置かない。永続化を Obsidian にするのは `~/workspace/tasks/` が削除されうるため（`tasks/CLAUDE.md` 規範に整合）。中間作業物のみ `~/workspace/tasks/<slug>/` 配下に置く。呼び出し時に明示パスがあればそれで上書きする。
- **出力ファイル名（basename）規約**:
  - 既定 basename: `<YYYY-MM-DD> skill-review-<対象 skill 名>.md`（日付プレフィックスで時系列ソート、skill 名で衝突回避）。
  - 同日同 skill の再評価で既存ファイルがある場合は時刻サフィックス `<YYYY-MM-DD>T<HH:mm> skill-review-<対象 skill 名>.md` を付ける（上書きを避ける）。
  - 呼び出し時に明示 basename があれば優先する。

## ガード

- 評価対象のスキル本体・委譲先を**変更しない**。read-only の評価に徹する（改善の適用は別タスク）。
- フェーズ2（実走）を**ユーザ承認なしに自動実行しない**（実装系スキルの起動・コスト・暴走防止）。
- 各指摘は**根拠（`SKILL.md:行` か trace イベント）併記が無ければ不採用**（grounding 必須）。
- 評価軸は3つ（フェーズ設計・委譲構造 / プロンプト・指示の品質 / 失敗・抜け穴の堅牢性）に固定する。トークン／コスト効率、委譲先スキル単体の改善、常設インフラ化は対象外。
- トレースが少ないうちは、フェーズ1(b) を統計的事実のように書かせない（1件のケーススタディとして位置づける）。
- **対象は自作 skill に限る**（ビルトイン skill・plugin skill は SKILL.md 不在で評価の足場が立たないため評価不能）。
- **引数なし／不在 skill 名で起動された場合は早期に停止する**（ステップ 0 のバリデーション。後続フェーズに進めない）。

## やってはいけないこと

- 評価対象や委譲先のファイルを編集する（read-only 評価のはずが実装に踏み込む）。
- 根拠なしの指摘を載せる（hallucinated line・牽強付会な引用。指摘ごとに根拠箇所を実際に開いて支持されるか確認する）。
- 行番号だけ引いて中身を確かめずに採用する／トレースを実読せず事象を主張する（逐語引用を必須にし、高・トレース由来は悉皆再照合する）。
- **評価対象 skill と同型の多段オーケストレータに肥大させる**（評価対象の複雑性を仕組み側に持ち込まない。読み取り系は1 subagent に統合する）。
- フェーズ2（実走）を承認なしに自動で回す。
- **精読不可の委譲先**を黙って評価対象から落とす、または精読したかのように書く（評価範囲の穴はレポートに明記する）。自動抽出で不在判定された委譲先（ビルトイン skill・plugin skill）も同じ扱い。
- 改善点レポートを `~/workspace/tasks/` 配下だけに残す（tasks は削除されうる。永続化は Obsidian）。
