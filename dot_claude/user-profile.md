# ユーザプロフィール（協働相手）

協働相手であるユーザの人物像・判断軸。`persona.md`（アシスタント側の態度）と対になる基盤文脈で、応答の framing と提案の評価軸に常時効かせる。ここに置くのは**安定した属性・判断軸**のみ。可変な個別状態（進行中の移行・採否の個別事例・マシン構成）は auto memory 側に置く。

## 職能と framing

- **職業ソフトウェアエンジニア。** エンジニアに通じる精度・具体で書く。
- 気づき・レポート・説明は、**ソフトウェア開発に転用できる形を最低1つ接地させる**と活用されやすい（デバッグ・依存管理・コード生成・設定移植・可観測性などの語彙で「次の開発でどう効くか」を示す）。トピックを限定する話ではなく framing の話。

## 技術倫理・法的境界

- 「技術的に**できる**」と「**やっていい**」を明確に区別する判断軸を重視する。公開本番システムへの無認可アクセス、サードパーティクライアントによる注文投入などの境界越えを「アウト」と即断し、「老害扱い」されてもこの線を下げない側に立つ。ハッカー文化の倫理コード（規則破りと自制の両立）を理解している層。
- 倫理・法的論点では、曖昧な留保や両論併記より**線がどこにあるかを明示する**応答を好む。思考が鋭く、こちらの留保を逆算して突いてくるので、**留保の根拠は最初から開示**しておく。

## OSS ライセンス・採否の判断軸

- ツール・ライブラリ採否ではライセンス形態を厳密にチェックする。根本軸は**「認知負荷の低さ」**——採否のためにライセンス文書を熟読・解釈する／enforcement への対応を考えるといった検討コスト自体を避けたい。
- 独自（非標準）ライセンス・商用利用禁止条項・法的に曖昧な条項・MDM 検出等の技術的 enforcement 機構を持つものは、**候補から外さず必ずアラート**（ライセンス名・商用可否・enforcement の有無を明示）してユーザの判断材料にする。最終判断はユーザに委ね、提案側で除外しない。標準ライセンス（MIT / Apache-2.0 / BSD / GPL 系など OSI/SPDX 標準）はアラート不要。

## シェル / CLI / ツール環境の判断軸

- **構造化を第一義にしない。** 構造化オブジェクト自体は否定しないが、それを活かすために eza/fd/rg/bat 等のモダン外部コマンドを捨てる羽目になるトレードオフ（PowerShell/nushell 型の内部コマンド縛り）を嫌う。
- 対話シェルには**モダン外部コマンドを素直に呼べること**を最優先。コマンド語彙は Unix 風を好む（`Get-Verb-Noun` 形式を嫌う）。「Unix 寄り」と「POSIX/bash 互換」は別概念として区別する。
- **複雑処理はシェル独自言語で頑張らず Python に外注する**派。対話とスクリプトを切り分ける。
- **Windows ネイティブ親和性を重視**（MSYS/Git Bash のパス変換摩擦を嫌う古参）。
- 評価は単一軸（語彙が Unix か等）に潰さず**多面で**見る。
